# 6.2 变量不是盒子：把“引用模型”的直觉建立起来（彻底告别共享可变对象的隐形坑）

## 一、核心认知：变量是「标签」，不是「盒子」

### 1.1 盒子模型为什么会把你带沟里

很多新手（甚至很多写了多年的人）潜意识里都是“盒子模型”：

- `a = [1, 2, 3]`：把列表“装进”变量 `a`
- `b = a`：把 `a` 盒子里的东西“复制一份”给 `b`

这个模型最致命的问题是：它无法解释**别名**。

### 1.2 标签模型：更贴近 Python 的真实运行方式

标签模型的说法是：

- 右侧先创建/得到一个对象
- 左侧名字只是“贴标签”，让这个名字指向该对象

所以：

- `b = a` 不会复制对象
- 它只是让 `b` 指向 `a` 当前指向的那个对象

你可以把下面这句话当成本节“唯一需要背下来的句子”：

> **赋值不会复制对象；赋值做的是“名字绑定”。**

---

## 二、别名（aliasing）：`b = a` 的真实含义

### 2.1 最小证据：改一个，另一个也变

```python
a = [1, 2, 3]
b = a
a.append(4)
print(b)  # [1, 2, 3, 4]
```

解释（按标签模型）：

- `a` 和 `b` 是同一个列表对象的两个名字
- 你修改的是“列表对象本体”，所以两个名字看见的内容都变了

### 2.2 再补一条关键证据：`id` 与 `is`

```python
id(a) == id(b)  # True：同一对象的身份标识
a is b          # True：同一对象
```

注意：

- `is` 比较的是对象身份（是不是同一个对象）
- `==` 比较的是值是否相等（内容是否一样）

这一点会在 6.3 里进一步展开。

---

## 三、赋值语句的本质：永远“先右后左”

你可以把赋值 `x = expr` 理解成两步：

1. **先算右边**：把 `expr` 求值（创建对象/得到对象）
2. **再绑定左边**：让名字 `x` 指向右边得到的对象

这不是“习惯”，而是你排查很多 bug 的依据。

---

## 四、关键实验：即使右边报错，对象也可能已经创建（Gizmo 证明法）

下面这个例子很适合用来“打碎盒子模型”，因为它能证明：

- 对象可以先出现
- 但如果右侧表达式整体失败，左侧名字不会绑定成功

```python
class Gizmo:
    def __init__(self):
        print(f"Gizmo id: {id(self)}")

x = Gizmo()
y = Gizmo() * 10
```

你会看到：

- `Gizmo()` 的 `__init__` 会先打印（说明对象确实创建过）
- 之后 `* 10` 报 `TypeError`
- 变量 `y` 不会被绑定（因为赋值的第二步没发生）

结论：

> **对象先于变量存在；变量不是对象的容器，只是名字绑定。**

---

## 五、术语纠正：用“绑定/引用”替代“分配/存储”

为了不让自己回到盒子模型，建议养成这套说法：

- 不说“把值分配给变量”，说“把名字绑定到对象”
- 不说“变量存储对象”，说“变量引用对象”
- 不说“复制给另一个变量”，说“创建别名/共享引用”

这是“语言习惯”，但它会影响你写代码时的设计判断。
