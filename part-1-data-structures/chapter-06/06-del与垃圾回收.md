# 6.6 `del` 和垃圾回收：`del` 删的是“名字绑定”，对象何时释放取决于可达性

这一节要把 Python 的内存管理讲到你能**解释清楚“对象为什么还活着/什么时候会死”**的程度。你会把下面这句话变成直觉：

> **`del x` 不是“删对象”，而是“把名字 x 和对象解除绑定”。**

对象能不能被回收，取决于它是不是还“可达”（是否还有强引用链能找到它）。

## 一、`del` 的本质：删除名字绑定（引用），不是删除对象

### 1.1 `del` 是语句，不是函数

`del` 的行为可以用“名字绑定”来理解：

- 变量名（如 `a`）只是标签
- `del a` 的意思是“把标签 `a` 撕掉”

撕掉一个标签不等于对象立刻消失，因为对象可能还有其他标签（别名）指向它。

### 1.2 最小证据：删掉 `a`，对象仍然活着（因为 `b` 还引用它）

```python
a = [1, 2]
b = a
del a
print(b)  # [1, 2]
```

解释：

- `del a` 只是把 `a` 这个名字删掉
- 列表对象仍然被 `b` 引用，因此仍可达、仍存活

当你再把 `b` 重新绑定到别处：

```python
b = [3]
```

如果这是最后一个强引用，那么旧的 `[1, 2]` 就变得不可达，才有机会被回收。

## 二、Python 的回收机制：引用计数 + 循环垃圾回收（GC）

你只需要把 Python 的回收理解成“两条腿走路”：

### 2.1 引用计数（CPython 的核心机制）

在 CPython 中，大多数对象的回收非常“即时”，因为有引用计数：

- 引用计数变为 0 → 对象立刻可释放（通常立即释放）

这解释了为什么很多情况下你一 `del` 最后一个引用，对象就立刻“死了”。

但引用计数有一个先天缺陷：**它处理不了循环引用**。

### 2.2 循环引用：引用计数的盲区

例如：

```python
a = []
b = [a]
a.append(b)
```

这时：

- `a` 引用 `b`
- `b` 引用 `a`

即使你把外部名字都 `del` 了，这两个对象仍然互相引用，引用计数不为 0。

为了解决这个问题，Python 还有一套 **循环垃圾回收器**（`gc` 模块），它会：

- 通过“可达性分析”找出不可达的循环结构
- 统一回收它们

### 2.3 `gc.collect()` 的意义

多数时候你不需要手动回收，但当你做教学/排查时：

- `gc.collect()` 可以强制触发一次循环回收
- 让“对象什么时候被清理”变成可观测证据

## 三、`__del__` 的误区：它不是“可靠析构函数”

很多语言有“确定性析构”（对象离开作用域必然销毁），但 Python 不是这么承诺的。

关键点：

- `__del__` 的执行时机不应被业务逻辑依赖
- 和循环引用混在一起时更麻烦
- 更推荐的资源管理方式是 `with`（上下文管理器）/ `try/finally`

结论：

> **不要把关键资源释放寄希望于 `__del__`。**

（本节 demo 会用 `weakref.finalize` 展示一种更安全的“对象结束时回调”的手段。）

## 四、弱引用：不增加引用计数，不阻止对象被回收

弱引用的核心一句话：

> **弱引用不会让对象“多活一会儿”。**

### 4.1 `weakref.finalize`：对象真正不可达时触发回调

你可以把它当成“对象生命周期观察器”：

- 你注册一个回调
- 对象被回收时（不再可达时）回调执行

这特别适合：

- 做缓存（对象不用了就让它自动消失）
- 做调试（确认对象到底有没有被释放）

### 4.2 弱引用的典型应用：弱键字典缓存

`weakref.WeakKeyDictionary` 的含义是：

- key 是弱引用
- key 对象一旦被回收，对应条目自动消失

这能避免“缓存把对象永远钉死在内存里”。

---

## 五、排查内存问题的入口（只给你最实用的）

当你怀疑“对象怎么还活着”：

- `gc.collect()`：强制回收一次（看回收数量）
- `gc.get_referrers(obj)`：谁在引用它（谨慎使用，会返回很多内部结构）
- `tracemalloc`：追踪内存分配的来源（更适合定位增长点）

这些工具不是日常写业务必用，但在排查泄漏时非常关键。
