## 一、同一性 vs 相等性：你到底在比较什么

### 1.1 `is`：比较“是不是同一个对象”

`a is b` 只回答一个问题：

> **`a` 和 `b` 指向的是不是同一个对象？**

它比较的是对象身份（identity），不可重载，速度很快。

在 CPython 里你可以把它近似理解为：

```python
id(a) == id(b)
```

（注意：`id` 的具体含义是实现细节，但“对象生命周期内唯一”这点你可以依赖。）

### 1.2 `==`：比较“值是否相等”（调用 `__eq__`）

`a == b` 关心的是“内容是否一样”，它通常会触发：

- `a.__eq__(b)`（或者 `b.__eq__(a)` 的回退逻辑）

因此：

- 它**可能很慢**（比如比较两个巨大的嵌套列表）
- 它**可能被重载**（这就是为什么不建议用 `x == None`）

## 二、别名（aliasing）：同一个对象的多个名字

### 2.1 经典例子：字典别名

```python
charles = {"name": "Charles L. Dodgson", "born": 1832}
lewis = charles            # 别名：同一个 dict 对象

print(lewis is charles)    # True：同一性

lewis["balance"] = 950     # 原地修改 dict
print(charles)             # charles 也“变了”
```

解释：

- `lewis = charles` 没有创建新字典
- 它只是在同一个对象上多贴了一个标签
- 所以你改的是“字典对象本体”，两个名字都看到变化

### 2.2 相等但不同一：值一样，不是同一个对象

```python
alex = {"name": "Charles L. Dodgson", "born": 1832, "balance": 950}
print(alex == charles)     # True：值相等（dict 按内容比较）
print(alex is charles)     # False：不是同一个对象
```

这两段是理解“同一性 vs 相等性”的最小对照。

## 三、`is` vs `==`：怎么选（以及最重要的硬规则）

### 3.1 `None` 判断必须用 `is`

你经常会看到两种写法：

- `x is None`（推荐）
- `x == None`（不推荐）

原因不是“风格问题”，而是**语义安全**：

- `is` 不可重载：你得到的是“确实是不是 None 对象”
- `==` 可能被重载：某些对象会让 `x == None` 产生奇怪结果

因此硬规则是：

> **判断是否为 `None`：永远写 `x is None` / `x is not None`。**

### 3.2 业务值比较用 `==`

如果你关心“值是否一样”，就用 `==`。

例如列表、字典、字符串、数值：

- `a == b`：关心内容
- `a is b`：关心是不是同一份对象（通常不是你要的）

### 3.3 判断是否别名用 `is`

当你怀疑“是不是共享了同一个可变对象”时，`is` 是最直接的证据：

- `a is b` 为 True：你确实有别名共享
- `a is b` 为 False：它们不是同一个对象（但里面的子对象可能仍共享，这是浅拷贝/深拷贝的主题）

## 四、元组的“相对不可变性”：元组本身不变，但元组里的可变对象会变

### 4.1 经典对照：`t1 == t2` 会因为内部列表变化而改变

```python
t1 = (1, 2, [30, 40])
t2 = (1, 2, [30, 40])

print(t1 == t2)      # True：初始相等

t1[-1].append(99)    # 合法：修改的是内部 list 对象
print(t1)            # (1, 2, [30, 40, 99])
print(t1 == t2)      # False：不再相等
```

这说明：

- 元组“不允许你替换槽位上的引用”（`t1[-1] = ...` 会报错）
- 但不阻止你修改“槽位引用指向的可变对象”

### 4.2 为什么包含可变对象的元组不可哈希

哈希要求“对象在生命周期内哈希值稳定”。如果元组里有可变对象：

- 元组的“逻辑内容”可能变化
- 哈希就不稳定

因此：

```python
hash((1, 2, [3]))  # TypeError: unhashable type: 'list'
```

这不是限制你“不能把 list 放进 tuple”，而是防止你做出“无法保证正确性”的哈希容器用法（如把它当 dict key / set 元素）。

## 五、补充：`id()` 的位置（理解即可，不要沉迷）

- `id(obj)` 在对象活着时唯一
- 对象销毁后，`id` 的整数值可能被复用

日常代码里很少需要直接用 `id()`：

- 判断同一性用 `is`
- 真要排查共享引用/别名问题，`id` 是好工具
