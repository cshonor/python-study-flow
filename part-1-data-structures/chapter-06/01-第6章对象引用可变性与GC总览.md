# 第 6 章开篇导读：对象引用、可变性和垃圾回收（把“隐形 bug”的根挖出来）

这一章要解决的不是“语法怎么写”，而是一个更底层、更决定代码质量的问题：

> **Python 里的变量不是盒子，是标签（名字）；对象才是数据本体；赋值只是让标签指向对象。**

如果你把这句话真正变成直觉，你会立刻发现很多“看起来诡异”的行为其实完全合理；反过来，很多隐形 bug 也是从“把变量当盒子”开始的。

## 一、变量、对象、引用：你必须先分清这三件事

### 1.1 “变量是标签，不是盒子”到底是什么意思

看这段最小例子：

```python
a = [1, 2, 3]
b = a
```

“错误直觉”（把变量当盒子）会让你以为：

- `a` 里装着列表
- `b = a` 是把列表复制给 `b`

“正确直觉”（名字是标签）是：

- 先创建了一个列表对象（它在内存里某个地方）
- `a` 这个名字指向它
- `b = a` 只是让 `b` 也指向同一个对象

因此：

- 你改列表对象（例如 `a.append(4)`），`b` 看到的是同一个对象，自然也“变了”
- 你把 `a` 重新绑定到另一个对象（例如 `a = []`），不会影响 `b`，因为 `b` 仍指向旧对象

### 1.2 `id()`、`is`、`==`：三个经常被混用的东西

你需要把它们拆开记：

- **`id(x)`**：对象的“身份标识”（在 CPython 中常见为内存地址的某种表示）。它用来说明“是不是同一个对象”。
- **`x is y`**：比较身份（identity），等价于比较 `id(x) == id(y)` 的语义。**只关心是不是同一个对象**。
- **`x == y`**：比较值（equality），通常会调用 `x.__eq__(y)`。**只关心内容是否相等**。

典型结论：

```python
a = [1, 2]
b = [1, 2]

a == b      # True：内容相同
a is b      # False：不是同一个列表对象
```

什么时候必须用 `is`：

- 检查是否为 `None`：`x is None`
- 判断是否确实是同一个对象（缓存、单例、哨兵对象等）

什么时候应该用 `==`：

- 你关心“值是否一样”，而不是“是不是同一份对象”

## 二、可变性（mutability）：到底什么东西会“原地改变”

### 2.1 核心结论：可变 / 不可变是**类型语义**，不是你能随手改的开关

在 Python 里，**每种内置类型在实现里就定好了**：这类对象的 API 是否允许**原地**改内部状态。你平时说的「改了 `a`」，经常是**名字重新绑定到另一个对象**；和「**同一个对象**在原地被改掉」不是一回事。

**内存直觉（教学模型，非 C 源码逐行）**

- **不可变类型**：对外表现为**没有**“在原对象上改值”的操作；像 `str`、`int` 这种，常见做法是**新造一个对象**，让名字改指向它（例如 `s = s + "!"` 里的 **`+` 产生新 `str`**）。  
- **可变类型**：对象内部有一块**可增长 / 可改写**的缓冲（如 `list` 的动态数组），**`append`、`__setitem__`** 等会在**原对象**上动手脚，**`id(list)` 往往不变**。

**为什么语言要分两派？（设计取舍）**

| 取向 | 不可变更擅长 | 可变更擅长 |
| :--- | :--- | :--- |
| **性能 / 语义** | **稳定哈希** → 可当 **`dict` 键、`set` 元素**；CPython 还有**小整数、字符串驻留**等复用（**实现细节**，勿依赖写业务） | **原地增删**：若每次改都复制整结构，像 `append` 成千上万次会变成**不可接受的开销** |
| **安全 / 别名** | 共享引用时，**不会在别处被偷偷原地改掉**（仍要注意**重新绑定**与复合操作的原子性） | 灵活表达「容器就是工作区」：`list`、`dict` 当累加器、缓存 |

### 2.2 常见内置类型一览

| 分类 | 类型 | 要点 |
| :--- | :--- | :--- |
| **不可变（一般语义）** | `int`、`float`、`bool`、`str`、`tuple`、`frozenset`、`bytes` | 不提供“原地改值”的 API；`tuple` 见下节「相对不可变」 |
| **可变** | `list`、`dict`、`set`、`bytearray` | 原地增删改；**不可哈希**（`set` 本身也不能进别的 `set`） |

### 2.3 “原地改变”到底指什么？

- **`list.append`**：通常**仍是同一个** `list` 对象（`id` 不变），长度和内容变。  
- **`s = s + "!"`**：产生**新** `str`，`s` 改绑到新对象（**旧 `str` 若无人引用可被回收**）。  
- **`a += 1`（`a` 为 `int`）**：绑定到**新的**整型对象（小整数可能 `id` 恰好复用，仍是实现细节）。

### 2.4 元组的“相对不可变性”：元组不变，但元组里的可变对象会变

```python
t = (1, [2, 3])
t[1].append(4)  # 合法
```

为什么合法？

- 元组“不允许你替换某个位置上的引用”：`t[1] = ...` 不允许  
- 但 `t[1]` 指向的那个列表对象本身是可变的，你改变的是列表对象的内部状态  

后果：**包含可变元素的元组通常不可哈希**（哈希须随对象语义稳定）。

### 2.5 自定义“像不可变 / 像可变”的类

```python
class ImmutablePoint:
    def __init__(self, x: float, y: float) -> None:
        self._x = x
        self._y = y

    @property
    def x(self) -> float:
        return self._x

    @property
    def y(self) -> float:
        return self._y


class MutablePoint:
    def __init__(self, x: float, y: float) -> None:
        self.x = x
        self.y = y
```

工程里更常见的「值对象不可变」写法是 **`dataclasses.dataclass(frozen=True)`**（见原书后续与 **`07-不可变类型技巧.md`**）。

### 2.6 用 `id()` 看：谁在原地改，谁在换对象

```python
lst = [1, 2, 3]
before = id(lst)
lst.append(4)
assert id(lst) == before  # 仍是同一个 list 对象

s = "hi"
before_s = id(s)
s = s + "!"
assert id(s) != before_s  # 新的 str 对象
```

**可运行对照**：`03_identity_equality_aliasing_demo.py` 第 **8)** 节（与本节对照阅读）。

**一句话**：**可变 / 不可变写在类型的行为里**；不可变换安全与哈希，可变换原地修改的效率；**`id()`** 帮你分清「原地改」和「换对象再绑定」。

## 三、浅拷贝 vs 深拷贝：拷贝的不是“值”，而是“引用结构”

你会在很多 bug 里看到“我明明 copy 了，怎么还会互相影响？”——原因通常是你做的是浅拷贝。

### 3.1 浅拷贝（shallow copy）

浅拷贝会：

- 创建一个新容器对象
- 但容器里面放的仍然是**旧元素对象的引用**

因此：

- 如果容器元素是不可变对象，浅拷贝通常没问题
- 如果容器元素是可变对象，两个容器会共享这些可变元素，从而互相影响

常见浅拷贝方式：

- `copy.copy(x)`
- `x[:]`（对序列）
- `list(x)` / `dict(x)`（在很多情况下也是浅拷贝语义）
- `x.copy()`（若类型提供，如 `list` / `dict` / `set`）

更完整的「背诵速查 + 最小例子 + 赋值/浅/深对照」见 [`04-浅拷贝为默认.md`](04-浅拷贝为默认.md)。

### 3.2 深拷贝（deep copy）

深拷贝会递归复制内部结构，尽量让所有层级都独立。

常见方式：

- `copy.deepcopy(x)`

但要知道深拷贝不是“无成本魔法”：

- 更慢、更占内存
- 遇到循环引用结构需要特殊处理（`deepcopy` 内部有备忘录机制）
- 对某些对象（文件句柄、锁、socket）深拷贝根本没有意义或会失败

所以结论是：

> 先想清楚“你要隔离到哪一层”，再选择浅/深拷贝；不要默认 deep copy。

## 四、函数参数：Python 是“共享传递”（call by sharing），不是“引用传递/值传递”二选一

最容易理解的一句话：

> **函数拿到的是对象引用的副本（名字绑定到同一个对象）。**

因此：

- 你在函数里**原地修改可变对象**（例如 `.append()`、`d[k]=...`），调用者也会看到变化
- 你在函数里把参数名重新绑定到新对象（例如 `x = []`），不会影响调用者（只是函数内部的名字换了绑定）

这也解释了“可变默认值”为什么是灾难。

## 五、可变默认值：最经典、最致命的“隐形 bug”

### 5.1 为什么会出 bug

```python
def f(a=[]):
    a.append(1)
    return a
```

关键事实：

> **默认值在函数“定义时”就计算一次，并被保存下来；之后每次调用复用同一个对象。**

所以：

- 第一次调用返回 `[1]`
- 第二次调用返回 `[1, 1]`
- 第三次调用返回 `[1, 1, 1]`

### 5.2 正确写法（通用模板）

```python
def f(a=None):
    if a is None:
        a = []
    a.append(1)
    return a
```

你会在 dataclass 里看到同样的原则：可变默认值要用 `field(default_factory=list)`。

## 六、`del` 与垃圾回收：`del` 删除的是“名字绑定”，不是“对象本体”

### 6.1 `del` 的真实语义

```python
del x
```

这表示：

- 把当前作用域中的名字 `x` 删除（解除绑定）
- 如果这是对象的最后一个强引用，才可能触发对象被回收

### 6.2 Python 的回收策略（你需要知道的层级）

你只要记住两个层面：

- **引用计数**：对象被引用一次计数 +1，解除引用计数 -1；当计数为 0，对象可立即回收（CPython 中常见）
- **循环垃圾回收（GC）**：为了解决“循环引用导致引用计数永不为 0”，Python 有额外的循环检测与回收（`gc` 模块）

你在排查“内存泄漏/对象活得太久”时，才需要更深入地看 `gc`。

### 6.3 `__del__`（析构方法）为什么不推荐

它的存在会让“对象何时被回收”变得更不可控，并可能与循环引用纠缠，带来难以调试的问题。除非你非常清楚自己在做什么，否则不要把关键逻辑放在 `__del__`。

## 七、弱引用（weakref）：缓存与映射的“保命符”

弱引用的核心点只有一句话：

> **弱引用不会增加引用计数，不会阻止对象被回收。**

适用场景：

- 缓存：你想“能复用就复用”，但不想“为了缓存把对象永远留住”
- 关联映射：`WeakKeyDictionary` 让 key 对象消失后，映射项自动消失

## 八、不可变对象的优化：为什么有时 `is` 看起来“也能用”

你会在 CPython 里见到：

- 小整数缓存（常见范围：`-5` 到 `256`）
- 字符串驻留（某些短字符串/标识符复用）

这会造成：

```python
a = 256
b = 256
a is b  # 可能是 True（实现细节）
```

但你必须把它当成实现细节，不要把业务逻辑建立在这种“恰好复用”的行为上：

> **判断值相等用 `==`；只有判断身份才用 `is`。**
