## 一、5.7 数据类导致的代码异味（Code Smell）

### 1.1 “代码异味”到底是什么

“代码异味”不是 bug：代码可能能跑、也可能测试都过了，但它暴露了一个事实：

> **这个设计未来会越来越难改、越来越难加需求、越来越容易引入新 bug。**

理解它的最好方式是：把它当成“设计层面的烟雾报警器”。

### 1.2 数据类最常见的两类异味

#### 异味 A：只有数据，没有行为（贫血模型 / Anemic Domain Model）

症状：

- 你有一堆 dataclass：`Order`、`User`、`Invoice`……
- 但它们几乎没有方法，只是字段集合
- 真正的业务逻辑全部散落在外部函数里：`process_order(order)`、`calc_tax(order)`、`validate_user(user)`……

为什么这是异味：

- **逻辑分散**：同一对象相关的规则散落到很多文件/函数里，很难“从对象入口找到规则”
- **容易复制粘贴**：不同地方写出“差不多的校验”，长期会分叉
- **封装被破坏**：对象的状态与不变量（invariant）不能由自己维护，外部随便改字段就能让对象进入“非法状态”

什么时候它不是问题：

- 你明确把 dataclass 当作 **DTO/VO**（数据传输对象 / 值对象），它只是中间表示：比如 API JSON、数据库行、CSV 行
- 这类对象的使命就是“搬运数据”，不承诺复杂行为

如何改：

- **把最核心的不变量/校验挪回类里**（常常放在 `__post_init__` 或显式方法中）
- 把“对外的行为”也放回类上，让调用点从：
  - `total = calc_total(order)`
  - 变成：`total = order.total()`

#### 异味 B：构造参数越来越长（Long Parameter List）

症状：

- dataclass 字段越来越多（5、8、12 个……）
- 调用处越来越可怕：`Resource(id, title, creators, date, type, description, language, subjects, ...)`
- 你开始不敢随便加字段，因为要改一堆构造调用

为什么这是异味：

- **易传错**：位置参数错一位，运行时不一定报错，但数据全乱
- **扩展成本高**：新增字段要改所有调用处

如何改（常见手段）：

- **强制关键字参数**（推荐）：把长构造写成关键字传参（调用处更稳定、更可读）
- **分层建模**：把一坨字段拆成更小的值对象/子 dataclass（例如 `Address`、`Money`、`GeoPoint`）
- **提供替代构造方式**：如 `from_row(...)` / `from_json(...)` / `parse(...)` 类方法，减少“到处手写一长串参数”

## 二、5.8 模式匹配类实例：`match ... case` 读起来像“声明式 if/elif”

模式匹配的目标不是“取代 if/elif”，而是解决一类很常见的问题：

> 你要根据对象的结构（类型、字段、嵌套结构）写很多层判断与解包。

`match/case` 让这件事更像“写规则”，而不是“写分支拼图”。

下面我们只讲和“数据类/类实例”最相关的三种形态。

---

## 三、准备一个可匹配的数据类：`City`

我们用一个 `City` 数据类（字段刻意写得像现实业务）：

- `continent`：洲（例如 `"Asia"`）
- `name`：城市名
- `country`：国家/地区代码
- `population_m`：人口（百万）

你会在 demo 里看到：同样的数据，用三种类模式匹配能写出完全不同风格的逻辑。

---

## 四、类模式 1：关键字模式（Keyword Pattern）—— **最推荐、最像“读字段”**

### 4.1 语法直觉

```python
case City(continent="Asia"):
    ...
```

你可以把它读成人话：

> “如果这是一个 `City`，并且它的 `continent` 字段等于 `"Asia"`。”

### 4.2 为什么它适合数据类

- 你写的是字段名：IDE 可补全、可重构
- 你可以只关心部分字段，忽略其他字段
- 不依赖字段顺序（字段顺序改了也不影响关键字匹配）

### 4.3 常见用法：只匹配部分字段 + 绑定变量

```python
case City(continent="Asia", country=code):
    # code 会绑定到 country 字段的值
    ...
```

---

## 五、类模式 2：位置类模式（Positional Class Pattern）—— **依赖 `__match_args__` 的字段顺序**

### 5.1 语法直觉

```python
case City("Asia", name, country, pop):
    ...
```

这看起来像构造函数调用，但它不是在构造对象，而是在“按顺序解包并绑定”。

### 5.2 `__match_args__` 是什么

位置模式靠 `__match_args__` 决定“第 1、2、3、4 个位置对应哪个字段”。

- dataclass 会自动提供 `__match_args__`（通常就是字段顺序）
- 你也可以打印出来确认（demo 里会打印）

### 5.3 什么时候用位置模式

- 你就是想按字段顺序解包（像拆元组一样）
- 或者你在写一个很“结构化”的解析器，位置更紧凑

什么时候别用：

- 业务对象字段经常变化时（顺序变化会把匹配写坏）

---

## 六、类模式 3：序列模式 + 类型约束（不是“类模式”，但经常被拿来对比）

书里常见一个对比：不用 dataclass，也可以匹配“像记录一样”的序列，例如：

```python
record = ["Tokyo", 35.6895, 139.6917]
```

你可以这样写：

```python
match record:
    case [str() as name, float() as lat, float() as lon]:
        ...
```

这里的关键点是：

- `str()` / `float()` 是 **类型检查**
- `as name` / `as lat` 是 **绑定变量**

新手最容易写错的点：

- `case [str(name), float(lat), float(lon)]:` 这种写法**不对**（把它当成“构造 + 绑定”了）
- 正确姿势是用 `T() as var`：`float() as lat`

---

## 七、把模式匹配和数据类放在一起的“最佳实践”

### 7.1 优先关键字模式

业务数据对象通常字段多、变化也多，所以：

- **关键字模式**可读性最好
- 字段增删对匹配的破坏最小

### 7.2 对“未知结构”留一个兜底分支

```python
case _:
    ...
```

这是在说：

> “如果没有任何规则命中，就走兜底逻辑。”

对外部输入（API/文件）尤其重要，否则匹配不到时你会丢失错误信息。

### 7.3 模式匹配不是银弹

如果你发现 match 越写越长，很可能你正在把“行为”写在外面：

- 这时回到 5.7：**把行为收回类里**，让 `match` 只做“分派/路由”，不要做一堆业务规则。
