## 一、本章到底解决什么？

你需要存**结构化数据**时，Python 给了你 4 种工具：

1. `namedtuple`
2. `NamedTuple`（`typing.NamedTuple`）
3. `@dataclass`（**最推荐**）
4. `TypedDict`

本章就是告诉你：

**什么时候用哪个、哪个最稳、哪个别乱用。**

---

## 二、4 种工具一句话看懂

### 1. `collections.namedtuple`

- 轻量、只读、**不可变**
- 本质是**元组 `tuple`**
- 能用 `.字段名` 访问
- 适合简单记录；**新代码更推荐带类型的 `NamedTuple`**

### 2. `typing.NamedTuple`

- 带**类型提示**的 named tuple
- 依然只读、不可变
- IDE / 类型检查更友好
- **现代简单只读记录**的首选之一

### 3. `@dataclasses.dataclass`（现代官方常用）

- 真正的**数据类**
- 自动生成 `__init__`、`__repr__`、`__eq__` 等（按需还可扩展）
- 可**可变**；也可 **`frozen=True` 做成不可变**
- 支持默认值、方法、继承
- **多数工程场景默认选它**

### 4. `TypedDict`

- **不是类**，只是给 **dict 的形状**加类型标注
- 运行时仍是普通 `dict`
- 主要给**类型检查器**用
- 适合 **JSON / API** 等“本来就是字典”的数据

---

## 三、最清晰对比（新手必看）

### 1. `namedtuple` / `NamedTuple`

- 轻量  
- 不可变  
- 满足条件时可哈希（与 tuple 一致）  
- 不适合塞复杂业务逻辑  
- 字段不能就地改值  

### 2. `dataclass`（能力最全、默认首选）

- 现代、标准、通用  
- 可变 / 不可变（`frozen`）可切换  
- 支持方法、默认值、校验（如 `__post_init__`）  
- 适合业务模型、DTO、配置等  
- **工程里“要一个对象装数据”时优先想它**  

### 3. `TypedDict`

- 描述字典有哪些键、值什么类型  
- 适合 JSON 等 dict 场景  
- **不是**真正的实例类型  
- **不能**像类那样封装方法  

---

## 四、新手永远记住的选型规则

1. **简单只读小记录 → `NamedTuple`**（优于裸 `namedtuple`）
2. **正式项目、业务数据、配置、实体 → `@dataclass`**
3. **字典结构、JSON / 接口形状 → `TypedDict`**
4. **`namedtuple`**：维护老代码时仍会见到；**新代码**简单记录请用 **`NamedTuple`**

---

## 五、最实用一句话总结

- **只读小数据 → `NamedTuple`**
- **业务数据对象 → `dataclass`（默认首选）**
- **字典类型标注 → `TypedDict`**

## 六、附录（进阶一点就够）

### `dataclass` 常用开关（先记名字）

- **`frozen=True`**：实例不可变（更像值对象）
- **`order=True`**：生成比较 / 排序相关方法（注意排序语义）
- **`slots=True`（3.10+）**：省内存、限制动态加属性（可选）

### 工程里怎么放

数据类常集中放在 `models.py` / `schema.py` / `types.py`，好找、好复用。  
它是**结构化数据的表达**，不必把全部业务逻辑都塞进一个 dataclass（更多讨论见 `06`、`07` 两篇）。
