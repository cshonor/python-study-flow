## 零、新手总览（先读这几句）

### 核心一句话

**编码**就是：把 **`str` 文本**按**固定规则**翻译成 **`bytes` 字节**。  
同一段文字，换不同编码，生成的**字节可以完全不同**。有的编码**根本不认识**某些字符，会在 **`encode`** 时直接抛 **`UnicodeEncodeError`**（不是“先乱码”，而是**写不出**）。

### 乱码从哪来？

用 **编码 A** 写成的 **`bytes`**，必须用 **A**（或与之**等价且一致**的规则）**`decode`** 回去。  
换成别的编码，要么 **`UnicodeDecodeError`**，要么**不报错但解出一堆鬼画符**（假成功、真乱码）。

## 一、codec 到底是什么？

在 Python 语境里：

- **codec**：某一种“编码/解码”规则的实现（既能 encode，也能 decode）。
- 例如：`utf-8`、`latin-1`、`cp1252`、`gb2312`、`utf-16le` 都是 codec 的名字（严格说是编码名，对应的 codec 由 Python 提供）。

你用的时候不需要背“100 多种 codec 列表”，你需要的是：

1. 能识别**常见编码名**（尤其是 Windows/老系统里常见的那几个）
2. 能用证据（hex）验证：某个字符在某个编码下到底变成了哪些字节
3. 知道“不能编码”意味着什么（会报错，不是自动帮你变）

```python
# 同一字符，不同编码 → 不同 bytes（用 hex 当证据）
"A".encode("ascii").hex(" ")    # '41'
"A".encode("utf-8").hex(" ")    # '41'（ASCII 区常一致）

"中".encode("utf-8").hex(" ")   # 'e4 b8 ad'

try:
    "中".encode("ascii")
except UnicodeEncodeError:
    pass  # ascii 没有「中」的码位：不是乱码，是直接不能 encode

# decode 必须与「真实字节来源」一致，否则报错或鬼画符
raw = "北京".encode("utf-8")
raw.decode("utf-8")   # 正确
# raw.decode("latin-1")  # 往往不报错，但得到错文本（假成功）
```

完整「字符 × 编码」表见 **§二**、脚本 **`05_codecs_encodings_table_demo.py`** 第 **1)** 节。

## 二、核心对照表：同一字符，在不同编码下的 bytes（hex）差异

下面这张表不靠截图，你自己就能跑出来复现（见脚本）。理解它，你就能解释“为什么同样的文字在不同环境会乱码”。

### 长什么样？（轮廓示意）

**维度**：**一行 = 一个字符**；**一列 = 一种编码**；**格子 = 该字符在该编码下的字节**（**空格分隔的 hex**），或 **`*`**（该编码**写不出**这个字符）。

```text
                 │  ascii  │ utf-8      │ gbk        │ …（脚本里还有 latin-1 / cp1252 / gb2312 / utf-16le）
─────────────────┼─────────┼────────────┼────────────┼
  行：'A'        │   41    │ 41         │ 41         │
  行：'中'       │    *    │ e4 b8 ad   │ d6 d0      │
  行：'€'        │    *    │ e2 82 ac   │    *       │
```

上面三行数值与 **`05_codecs_encodings_table_demo.py`** 当前输出一致，便于你对照；**完整 6 字符 × 7 编码**仍以脚本为准。

**子表（与脚本第 1) 节一致，节选）**：

| char | ascii | utf-8 | gbk |
| :--- | :--- | :--- | :--- |
| `A` | `41` | `41` | `41` |
| `é` | `*` | `c3 a9` | `a8 a6` |
| `€` | `*` | `e2 82 ac` | `*` |
| `中` | `*` | `e4 b8 ad` | `d6 d0` |

**读法**：横向对比「同一字符、不同编码」；纵向对比「同一编码、不同字符」；**`*`** = **`UnicodeEncodeError`** 边界。

### 2.1 表格怎么读？

- 行：字符（例如 `A`、`é`、`€`、`中`）
- 列：编码（例如 `ascii`、`latin-1`、`cp1252`、`gb2312`、`utf-8`、`utf-16le`）
- 单元格内容：
  - 形如 `e4 bd a0`：表示编码后得到的 bytes（十六进制）
  - `*`：表示该编码**无法表示这个字符**（会抛 `UnicodeEncodeError`）

### 2.2 你应该从表里得到的结论

1. **ASCII 兼容**：很多编码对 `A-Z/a-z/0-9` 都一致（`A` 通常都是 `41`）。  
2. **非 ASCII 字符差异巨大**：`é`、`€`、中文在不同编码下会完全不同。  
3. **能不能表示，是“硬边界”**：比如 `ascii` 表示不了 `中`，不是“乱码”，而是直接不能编码。  
4. **UTF-8 是变长**：同一个字符可能 1 字节、2 字节、3 字节甚至 4 字节。  
5. **UTF-16LE 常见为 2 字节（BMP 内）**：但遇到更高码点（例如部分 emoji）会变成 4 字节（代理对）。

## 三、常见编码标准（新手够用版）

这一节不追求百科全书，只讲你在真实工程里会遇到的“高频编码”。

### 3.1 `latin-1`（ISO-8859-1）

- **特点**：单字节编码，能映射 0–255 的所有字节值到某个字符。  
- **工程意义**：
  - 它“几乎不会解码失败”（因为任何 byte 都能映射到某个字符），所以**很容易产生“看似成功、实际乱码”**。
  - 在排查时有时会用它做“保留原始字节值”的临时手段，但这不是正确解码。

### 3.2 `cp1252`（Windows-1252）

- **特点**：Windows 西欧代码页，很多环境里被误叫做“ANSI”。  
- **工程意义**：你在 Windows/旧数据里见到的“奇怪引号/欧元符号”等，常与它有关。

### 3.3 `gb2312` / `gbk`（中文相关）

- **`gb2312`**：较早的简体中文编码标准（覆盖范围更小）。  
- **`gbk`**：更常见、覆盖更广（现实里你更可能碰到它）。  
- **工程意义**：在 Windows 控制台、旧文件、旧接口里经常出现；你必须显式写 `encoding="gbk"` 才能对上。

### 3.4 `utf-8`（默认首选）

- **特点**：覆盖全部 Unicode，ASCII 兼容，变长编码。  
- **工程意义**：Web/现代系统最常用。你应当把“UTF-8 + 显式指定 encoding”当作默认策略。

### 3.5 `utf-16le`（需要认识，但别滥用）

- **特点**：常用于某些系统接口/文件格式；在很多字符上表现为 2 字节，但不是“永远 2 字节”。  
- **工程意义**：当你看到文本文件以 `FF FE` 或 `FE FF` 开头（BOM）时，就要警觉可能是 UTF-16。

## 四、你能立刻用上的排查套路（用 hex 说话）

当你遇到“乱码/异常”，不要凭感觉猜编码。你需要做的是：

1. **拿到原始 bytes**（文件就用二进制读：`Path(...).read_bytes()`）  
2. **打印前 64 字节 hex**：`raw[:64].hex(' ')`  
3. **尝试用候选编码 decode**，并用 `ascii()` 打印结果（防止控制台输出阶段再出坑）

配套的“多编码对照表脚本”就是把这件事做成了可视化：同一字符在不同编码下的 hex 一眼对比。

## 六、一页极简背诵版（口诀 + 要点）

### 三句口诀

1. **编码管翻译**：**`str.encode` → `bytes`**；**`bytes.decode` → `str`**；**同写同读**，别混编码名。  
2. **先看字节再猜**：**`read_bytes` + `hex(" ")` + `ascii(解码结果)`**，不靠蒙。  
3. **新项目默认 UTF-8**：**`encoding="utf-8"`** 写清楚；老中文环境才盯 **GBK**；**`latin-1`** 解码**几乎不失败**——**最容易假正常**。

### 五种编码「一句话」

| 编码 | 记住这一嘴就够 |
| :--- | :--- |
| **`latin-1`** | 单字节 0–255 全覆盖，**解码几乎总“成功”**，排查时**最像“解开了”的陷阱** |
| **`cp1252`** | Windows 西欧页，**弯引号 / 奇怪符号**常见来源 |
| **`gb2312` / `gbk`** | 中文本地老数据；现实里 **`gbk`** 更常碰到 |
| **`utf-8`** | **现代默认**；**Unicode 全覆盖**；**ASCII 兼容**；**变长** |
| **`utf-16le`** | 常 2 字节一字；**BOM `FF FE` / `FE FF`** 要警觉；部分 **emoji** 会到 4 字节 |

### 对照表四个结论（速记）

1. **英文/数字**：多数编码与 **ASCII** 对齐，**`41` 一类**常一致。  
2. **`é` / `€` / 中文**：各编码 **hex 完全可能不同**。  
3. **`ascii` 写不出中文**：是 **`UnicodeEncodeError`** 硬边界，不是“先乱码”。  
4. **UTF-8 变长**：同一串里可混 **1～4 字节**单元（以 **`len(ch.encode('utf-8'))`** 为准）。
