# 字符、码点、编码与错误处理（第 4 章入门）

这一篇只做一件事：把 **字符 / 码点 / 编码 / 字节** 这 4 个词的关系讲清楚，然后让你能稳稳处理两类常见异常：

- `UnicodeDecodeError`（bytes → str 失败）
- `UnicodeEncodeError`（str → bytes 失败，或输出/写文件时失败）

配套脚本：`02_codepoints_encoding_demo.py`（输出用 `ascii()`，避免 Windows 控制台编码差异干扰理解）。

---

## 一、四个词一张图（心智模型）

- **字符**：人类理解的符号，例如 `A`、`中`、`€`、`é`。  
- **码点**：Unicode 为字符分配的编号，例如 `A` 是 `U+0041`，`中` 是 `U+4E2D`。  
- **编码**：把「码点序列」翻译为「字节序列」的规则，例如 UTF-8、UTF-16、GBK。  
- **字节**：最终落在磁盘 / 网络 / 进程间传输的 0–255 数字序列。

在 Python 3 里，你只要牢记两句话：

- **`str` 是文本（码点序列）**：面向人类语义。  
- **`bytes`/`bytearray` 是字节序列**：面向存储与传输。

因此跨越边界的动作必须显式发生：

- **编码**：`str.encode(encoding)` → `bytes`  
- **解码**：`bytes.decode(encoding)` → `str`

---

## 二、`ord` / `chr`：字符与码点的桥

- `ord(ch)`：字符 → 码点整数  
- `chr(cp)`：码点整数 → 字符

你可以用它们把“看不见的码点编号”变成可验证的事实，例如把字符打印成 `U+....` 形式。

---

## 三、为什么 UTF-8 下「字节数」可能比「字符数」多

### 核心定义

1. **`len(s)`**  
   统计的是 **Unicode 码点个数**（日常口语里常叫「字符数」）：**一个 `str` 元素占一格**，与「这个字在 UTF-8 里占几字节」无关。

2. **`len(s.encode('utf-8'))`**  
   先把 **`str`** 按 **UTF-8** 编成 **`bytes`**，再数**字节个数**。

### 底层原因：UTF-8 是变长编码

常见分档（**具体以 `len(ch.encode('utf-8'))` 为准**）：

- **ASCII 区**（`U+0000`–`U+007F`）的英文字母、数字、常用符号：**1 字节 / 码点**。  
- **许多带重音符的拉丁字母**（如 **`é`**，预组合形式）：常为 **2 字节 / 码点**。  
- **大量中文、日文、韩文等**（常用 BMP 块）：多为 **3 字节 / 码点**。  
- **不少 emoji、部分增补平面字符**：多为 **4 字节 / 码点**。

只要串里出现 **「非 1 字节 UTF-8 单元」** 的码点，就可能出现 **`len(s.encode('utf-8')) > len(s)`**；**纯 ASCII** 时二者**常相等**。

### 直观代码示例

```python
s1 = "a"
len(s1), len(s1.encode("utf-8"))  # (1, 1)

s2 = "中"
len(s2), len(s2.encode("utf-8"))  # (1, 3)

s3 = "\U0001f600"  # 😀
len(s3), len(s3.encode("utf-8"))  # (1, 4)
```

### 一句话

**`len(s)`** 数的是**码点个数**；**`len(s.encode('utf-8'))`** 数的是**按 UTF-8 铺开后的字节占地**——变长编码下，**「一人」未必只占「一字节」**。

可运行对照：**`02_codepoints_encoding_demo.py`** 第 **2)** 节（含 **`café`** 与 **ASCII / 中文 / emoji** 分档）。

## 四、两类经典异常：`DecodeError` vs `EncodeError`

（Python 3 里具体类型名是 **`UnicodeDecodeError`** / **`UnicodeEncodeError`**。）

### 1) `UnicodeDecodeError`

把 **bytes** 按“错误的编码”解释为文本时失败。

典型场景：拿到一段 **UTF-8** 字节，却用**另一套编码**去 `decode()`——轻则**乱码**，重则**抛错**（取决于那段字节在对方规则下是否「合法」）。

```python
raw = "北京".encode("utf-8")  # 按 UTF-8 得到的字节
raw.decode("utf-8")          # '北京'，正确
raw.decode("gbk")            # 常见：不抛错，但得到乱码（本仓库脚本会打印 ascii(wrong)）
raw.decode("ascii")          # 一定 UnicodeDecodeError：ASCII 不允许 >127 的字节
```

### 2) `UnicodeEncodeError`

把 **str** 写成字节时，目标编码无法表示某些字符时失败。

```python
"北京".encode("utf-8")   # 正常
"北京".encode("gbk")     # 正常（另一套合法映射）
"北京".encode("ascii")   # UnicodeEncodeError：ASCII 里没有「北」「京」
```

典型场景：**str → bytes** 时目标编码字库不够大；例如 **GBK** 写不出 **emoji**，严格模式下直接报错：

```python
"😀".encode("gbk")     # UnicodeEncodeError：illegal multibyte sequence
"😀".encode("utf-8")  # 正常，得到 4 字节 UTF-8
```

另一典型场景：Windows 控制台的 **`sys.stdout.encoding`** 常为 **GBK**，**`print(含 emoji 的 str)`** 时，解释器要把文本**按该编码编成字节**再交给终端——若字符不在字库里，就会在**输出阶段**触发 **`UnicodeEncodeError`**（或退化为替换，取决于 **`PYTHONIOENCODING`** / **`errors=`** 等配置）。  
因此可能出现：**内存里 `str` 已经是对的**，一 **`print`** 才炸。仓库脚本用 **`ascii()`** 打印字符串，正是为了**绕开**这一层，先把「文本层」讲清楚（见 **§七**）。

## 五、`errors=`：当你必须“继续走下去”

`encode`/`decode` 都支持 `errors=`，常用策略：

- **`'strict'`**：默认，遇到问题直接抛异常（最安全）。  
- **`'replace'`**：用替换字符代替（解码常见为 `\ufffd`）。  
- **`'ignore'`**：直接丢弃无法处理的部分。

### 例子：`decode`（bytes → str）

```python
bad = b"\xff\xfe\x00"

bad.decode("utf-8", errors="strict")   # UnicodeDecodeError（省略 errors= 时默认如此）

ascii(bad.decode("utf-8", errors="replace"))  # 典型含 '\ufffd'（替换符），以实际输出为准

ascii(bad.decode("utf-8", errors="ignore"))   # 非法 UTF-8 片段被丢，常见只剩 "\x00"
```

### 例子：`encode`（str → bytes）

```python
s = "北京"

s.encode("ascii", errors="strict")   # UnicodeEncodeError

s.encode("ascii", errors="replace")  # b'??'

s.encode("ascii", errors="ignore")   # b''
```

策略选型取决于业务目标：日志/监控可能优先“不断流”，协议解析或数据清洗通常更偏向 `'strict'`（宁可失败也不要悄悄吞掉错误）。

## 六、实践原则（再次强调）

- **尽早解码，最晚编码**。  
- **文本文件读写显式 `encoding=`**：不要依赖平台默认编码。  
- 遇到乱码/异常时，优先定位“到底是哪一步在 `encode` 或 `decode`”，再决定修复编码参数或错误处理策略。
