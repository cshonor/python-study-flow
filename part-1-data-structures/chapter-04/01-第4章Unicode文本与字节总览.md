# 第 4 章「Unicode 文本与字节序列」开篇：先把 `str` / `bytes` 的边界讲清楚

Unicode/编码这块，新手最常见的痛点不是“概念多”，而是**边界不清**：什么时候你手里是文本（`str`），什么时候你手里是字节（`bytes`）？什么时候该 `encode`，什么时候该 `decode`？

这一章你只要先建立一个稳定的心智模型：**`str` 是人类文本（码点序列），`bytes` 是存储/传输用的字节序列**。两者之间的转换必须显式，并且要写清楚编码规则。

---

## 一、核心纲领

> 文本给人类阅读，字节序列供计算机处理。  
> ——Esther Nam 和 Travis Fischer（书中引用）

- **`str`**：Unicode **字符序列**（逻辑上是**码点**序列），**不绑定**某一具体「磁盘上的字节形态」。  
- **`bytes` / `bytearray`**：**字节**的不可变 / 可变序列，是「物理存储与传输」层面的数据。  
- **编码（encode）/ 解码（decode）**：在「文本」与「某套编码规则下的字节」之间**显式**转换；Python 3 **取消**了 Python 2 式 **`str`/`unicode` 与字节混用时的隐式转换**，从源头减少乱码类 bug。

---

## 二、本章知识地图（结构化）

| 模块 | 核心知识点 | 学习价值 |
| :--- | :--- | :--- |
| **字符、码点、字节** | 字符与 **Unicode 码点**；**UTF-8** 等编码的角色 | 读懂「为何同一字符会有不同字节」 |
| **二进制序列类型** | **`bytes`**、**`bytearray`**、**`memoryview`** | 网络、文件、缓冲区的高效处理 |
| **编解码与错误** | **`encode`/`decode`**；**`strict`/`ignore`/`replace`** 等 | 处理 `UnicodeEncodeError` / `UnicodeDecodeError` |
| **实践原则** | **尽早解码、最晚编码**；打开文本时显式 **`encoding`** | 跨平台、跨语言环境少踩坑 |
| **规范化与比较** | **NFC/NFD** 等；语义相同但码点序列不同的字符串 | 安全比较、搜索与存储 |
| **排序与本地化** | **`locale`**、**Unicode 排序**（书中可能提及 **`pyuca`** 等） | 按语言习惯排序，而非仅按码点 |
| **Unicode 数据与双模式 API** | **`unicodedata`**；**`os`/`re`** 等路径或模式对 **`str`/`bytes`** 的分支 | 系统边界与正则的编码意识 |

后续小节将按书内顺序在 `chapter-04/` 内逐篇展开（当前仓库以**本篇导读 + 入门 demo** 为起点）。

## 三、为什么要精读本章

- **高频问题**：爬虫与 HTTP、读写文件、数据库与 JSON、子进程与管道——凡涉及「字节进、文本出」之处，编码错误都会表现为**乱码**或**异常**。  
- **工程能力**：国际化文本、二进制协议、加密与压缩，都要求你**自觉**处理 **`str` 与 `bytes` 的边界**。  
- **平台差异**：**默认编码**依赖 **locale** 与 **Python 版本**（例如文本模式打开文件时的默认、标准流编码），**不要**在代码里假设「永远是 UTF-8」或「永远是某一种中文编码」；**显式指定**最稳。
