# I/O 编码排查清单：文件 / 网络 / 子进程 / 控制台（第 4 章实战）

编码问题最烦的点是：它看起来“随机”，但其实只要抓住 **类型** 和 **边界**，就能系统化排查。

这份清单的目标是让你遇到乱码/异常时不靠猜：

- 先确认你手里到底是 `str` 还是 `bytes`
- 再定位“哪一步跨过了边界”（文件/网络/数据库/子进程/控制台）
- 最后用最小证据（`hex` / `ascii()`）验证你的判断

配套脚本：`03_io_encoding_troubleshoot_demo.py`。

---

## 一、先问 3 个问题（90% 的问题在这里结束）

1. **你手里的是 `str` 还是 `bytes`？**  
   - `str`：文本（码点序列）  
   - `bytes`：字节（0–255）
2. **边界在哪里？**（哪一步把 bytes 变成 str，或反过来）  
   - 文件读写、HTTP、数据库驱动、子进程、标准输出
3. **你使用/假设的编码是什么？是否与真实来源一致？**  
   - 不要靠“猜默认”，靠**显式参数**或协议/元数据

---

## 二、最常见的坑与一眼定位

### 1) `TypeError: can't concat str to bytes`（或类似）

- **原因**：混用了 `str` 与 `bytes`，通常是边界没处理好
- **排查**：对关键变量 `print(type(x), repr(x)[:...])`
- **修复**：明确一端 `encode()` 或 `decode()`（并写清编码）

```python
"prefix: " + b"tail"  # TypeError: can only concatenate str (not "bytes") to str

# 正确拼在同一边界上（二选一思路）
"prefix: ".encode("utf-8") + b"tail"  # 全程 bytes
"prefix: " + b"tail".decode("utf-8")  # 全程 str
```

### 2) 不报错但“乱码”（mojibake）

- **原因**：用错编码 `decode()` 了，但“碰巧能解出来”
- **排查**：
  - 固定展示证据：**原始 bytes 的 hex**，以及 `ascii()` 形式的输出
  - 对同一段 bytes，用不同编码 `decode()` 对比
- **修复**：纠正真实编码来源；兜底策略只是止血，不是根治

```python
raw = "北京".encode("utf-8")
print(ascii(raw.decode("utf-8")))  # '\u5317\u4eac' 正确
print(ascii(raw.decode("gbk")))    # 常不抛错，但已是乱码（与脚本「3)」对照）
```

### 3) `UnicodeDecodeError`（bytes → str）

- **原因**：bytes 不是你以为的编码，或数据损坏/截断
- **排查**：
  - 打印 `hex` 头部：`raw[:64].hex(' ')`
  - 明确 bytes 来源：文件声明/HTTP header/数据库连接配置
- **修复**：用正确的编码 `decode()`；必要时使用 `errors=`（见 §五）

```python
raw = "北京".encode("utf-8")
raw.decode("ascii")  # UnicodeDecodeError：多字节 UTF-8 头字节 > 127

b"\xff".decode("utf-8")  # UnicodeDecodeError：非法 UTF-8 起始字节
```

### 4) `UnicodeEncodeError`（str → bytes，或 `print()` 时发生）

- **原因**：目标编码无法表示该字符；在 Windows 上常见于控制台输出阶段
- **排查**：看 `sys.stdout.encoding` 与 `sys.stdout.errors`
- **修复**：
  - 代码里：尽量不要依赖控制台默认编码；日志/调试可用 `ascii(s)` 或 `backslashreplace`
  - 环境上：使用 UTF-8 终端/启用 UTF-8 输出（视你环境而定）

```python
"北京".encode("ascii")  # UnicodeEncodeError（ASCII 无汉字码位）

"\U0001f600".encode("gbk")  # 常见：GBK 无 emoji，strict 下 UnicodeEncodeError
```

（**`print`** 阶段等价于把 **`str`** 按 **`sys.stdout.encoding`** 去 **`encode`**；字符不在该编码字库里时，就会在这一步失败。调试请优先 **`ascii(s)`** 或写入 **UTF-8 文件**再打开，见 **`02-码点编码与编解码错误.md`** **§四**、**§七**。）

## 三、文件 I/O：推荐模板（强烈建议复制粘贴）

### 文本读写（推荐）

```python
from pathlib import Path

text = Path("in.txt").read_text(encoding="utf-8", errors="strict")
Path("out.txt").write_text(text, encoding="utf-8", errors="strict")
```

### 二进制读写（推荐）

```python
from pathlib import Path

raw = Path("in.bin").read_bytes()
Path("out.bin").write_bytes(raw)
```

**要点**：

- 文本模式就显式 `encoding=...`；二进制模式就只处理 bytes  
- 不要在文本模式下手动 `.decode()`（除非你在做教学/实验）

## 四、子进程与标准输出：边界要显式

### 子进程拿到文本输出（推荐）

```python
import subprocess, sys

cp = subprocess.run(
    [sys.executable, "-c", "print('北京')"],
    text=True,
    encoding="utf-8",
    capture_output=True,
    check=True,
)
print(repr(cp.stdout))
```

### 控制台输出导致的“最后一公里”问题

Windows 上 `sys.stdout.encoding` 可能不是 UTF-8。为了让教学 demo **不因控制台编码崩溃**，可以：

- 用 `ascii(s)` 打印（只输出 ASCII 转义，稳定）  
- 或写入 UTF-8 文件再用支持 UTF-8 的工具打开

## 五、`errors=`：兜底策略（谨慎使用）

常见策略：

- **`'strict'`**：默认，异常直接抛出（推荐优先使用）  
- **`'replace'`**：用替换字符代替（可能造成信息丢失）  
- **`'ignore'`**：直接丢弃（信息丢失更严重）

建议：把 `errors=` 当作**降级策略**，并在日志里记录“降级发生过”。

### 例子：`decode`（bytes → str）

对一段**不是合法 UTF-8**的字节，三种策略对比（输出用 **`ascii()`** 便于在任意终端查看）：

```python
bad = b"\xff\xfe\x00"

bad.decode("utf-8", errors="strict")   # UnicodeDecodeError（默认即 strict）

ascii(bad.decode("utf-8", errors="replace"))  # '\\ufffd\\ufffd\\x00'（非法字节变成替换符）

ascii(bad.decode("utf-8", errors="ignore"))   # '\\x00'（能丢的 UTF-8 片段被丢掉）
```

**`Path.read_text(encoding=..., errors=...)`** 同样支持 **`errors=`**（与 **`open(..., errors=...)`** 一致）。

### 例子：`encode`（str → bytes）

当**目标编码写不出**某些字符时，`errors=` 决定是报错还是“硬写出去”：

```python
s = "北京"

s.encode("ascii", errors="strict")   # UnicodeEncodeError

s.encode("ascii", errors="replace")  # b'??'（用 ASCII 可打的占位符替代）

s.encode("ascii", errors="ignore")   # b''（无法表示的字符全部省略）
```

可运行对照见 **`03_io_encoding_troubleshoot_demo.py`** 第 **5)** 节（**`decode`** 与 **`encode`** 两侧）。

## 六、最小证据工具箱（排查时非常好用）

协议、抓包、文件头、编码排查时，手里往往只有 **`bytes`** 或难打印的 **`str`**。下面两件**不依赖控制台编码**，适合当作「最小证据」先固定下来。

### 1) 打印 `bytes` 的 hex 头（`hexdump`）

**作用**：截取二进制数据**头部**，用**空格分隔的十六进制**展示，和许多抓包/协议工具的习惯一致，**一眼能对齐字节边界**。

```python
def hexdump(b: bytes, n: int = 64) -> str:
    return b[:n].hex(" ")  # Python 3.8+：sep 为空格 → 每字节一对 hex，中间空格
```

- 默认只看前 **`n`** 字节（常用 **64**，避免日志被超长样本淹没）。  
- **不经过文本解码**，不会因为「不可打印字节」或乱码展示而误导判断。  
- 与 **`bytes.fromhex` / `bytes.hex`** 同一套 hex 视角；整段互转写法见 **`02-码点编码与编解码错误.md`** **§六**。

#### 为什么「先盯前 64 字节」就够用？（工程习惯）

- **BOM / 魔数 / 协议头**多在**流的最前面**：UTF-8 BOM、UTF-16 的 **`FF FE` / `FE FF`**、常见二进制头、甚至「一眼全 ASCII」的 hex 形态，都集中在**头部几十字节**里。  
- **同一条字节流**在常规场景下**通常只遵循一种文本编码**——不会前半 UTF-8、后半无故切成 GBK；**先读头**等于先确认「这条流自报的格式线索」。  
- **`raw[:64].hex(" ")`** 是**性价比**：少读盘、少刷屏，却能覆盖**绝大多数**「猜错编码 / 看错 BOM / 把二进制当文本」的第一轮误判。  

这不是数学定理：遇到**拼接容器、分段各自编码**等怪格式，仍要扩大范围或按协议拆帧；详见 **`05`**、**`06`** 与 **`02`**。  
**ASCII / Unicode / UTF-8 谁管什么**见 **`01-第4章Unicode文本与字节总览.md`** **§四**。

### 2) 安全展示 `str`（`ascii`）

**作用**：把 **`str`** 打成 **ASCII 可打印的转义形式**（**`\uXXXX` / `\UXXXXXXXX`**），在 **Windows / Linux / macOS** 各类终端上都**稳定、可复制**，减少 **`print` 触发的 `UnicodeEncodeError`** 或肉眼乱码干扰。

```python
print(ascii(s))
```

- 排查「**内存里的 `str` 到底是不是你以为的内容**」时，比直接 **`print(s)`** 更可靠。  
- 若希望封装成函数，一行即可：**`def safe_print(s: str) -> None: print(ascii(s))`**。

### 工具箱完整可复制版

```python
def hexdump(b: bytes, n: int = 64) -> str:
    return b[:n].hex(" ")


def safe_print(s: str) -> None:
    print(ascii(s))
```

### 小结

- **`hexdump(b)`**：先看 **`bytes`** 长什么样（**头部 hex**）。  
- **`ascii(s)` / `safe_print(s)`**：再看 **`str`** 长什么样（**转义**）。  
- 二者常一起用：先确认**字节侧**是否对，再确认**解码后的文本侧**是否对，是编码/网络/二进制问题里**出现频率很高**的一对组合。
