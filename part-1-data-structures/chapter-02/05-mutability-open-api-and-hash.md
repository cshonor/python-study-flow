# 可变 vs 不可变：到底是谁决定的？（协议视角 + 哈希）

> **一句话**：一个类型是「可变」还是「不可变」，取决于**语言/实现是否为「在同一对象身份下改内容」开放了接口**——在 Python 层常体现为**有没有**元素级写入、原地增删等**协议**（如 `__setitem__`、`append`）。  
> 这与 **「名字重新绑定到另一个对象」**（`a = a + 1` 对不可变类型常是**新对象**）不是同一回事。  
> 设计权衡与 ABC 见：`03-sequence-flat-container-mutable-immutable-abc.md`。

---

## 一、最直白：开放不开放「改自己」的入口

|  | **不可变**（`str`、`tuple`、`bytes`、`frozenset`、数字等） | **可变**（`list`、`dict`、`set`、`bytearray` 等） |
| :--- | :--- | :--- |
| **直觉** | 对象一旦建好，**不提供**「在同一身份下改内部状态」的常规路径 | **提供**原地修改、增删等操作 |
| **你想「改内容」** | 往往只能**新建**一个对象，再让名字绑过去 | 常可**原地**改，同一 `id` 下内容变 |

**注意**：`tuple` 里若嵌了 `list`，**元组槽位不能换**，但**内层列表仍可改**——不可变的是**元组这一层**，不是「所有下级对象」。

---

## 二、用 Python 行为一眼看穿：有没有「修改」入口

### 不可变：没有元素级写入

```python
s = "hello"
# s[0] = "x"  # TypeError：str 不支持按索引赋值
```

序列里 **`str` / `bytes` / `tuple`** 不提供**按索引的 `__setitem__` 语义**（你找不到「改某一位」的合法操作）。

### 可变：实现了写入与增删

```python
lst = [1, 2, 3]
lst[0] = 999   # OK：list 实现了 __setitem__
lst.append(4)  # OK：原地增长
```

**映射、集合**同理：`dict`/`set` 有 `__setitem__`/`add`/`discard` 等，允许在**同一对象**上改映射关系或成员。

---

## 三、和 CPython 实现怎么对上号（人话）

不必读 C 源码也能记：

- **不可变类型**：不在解释器/API 层给你「改这块对象体里已有内容」的常规路径；试图改的操作在 Python 层**抛错**或**隐式变成新建对象**（如部分 `str` 拼接实现可优化，但语义仍是新串）。
- **可变类型**：对象体里带**可增长/可改槽位**等结构（如 `list` 的指针数组与容量），并在类型上挂好了**增删改**的 C 实现与 Python 方法。

**一句话**：**类型设计者给不给你「在同一 `id` 上改内容」的权限**，决定你在 Python 里感受到的可变/不可变。

---

## 四、为什么要故意做成不可变？

1. **安全 / 清晰**：把不可变对象传出去，调用方**不指望**它在原地被改（仍要警惕**可变元素**）。
2. **可哈希**：内容不变 → `hash()` 可稳定 → 可作 **`dict`/`set` 的键**（须类型本身可哈希且元素可哈希等条件满足）。
3. **复用**：小整数、部分字符串等可被**驻留/复用**（实现细节），依赖**不变性**才安全。

---

## 五、和哈希串起来（复习）

|  | **不可变且可哈希** | **可变** |
| :--- | :--- | :--- |
| **内容** | 生命周期内不变（对该对象） | 可在原地变 |
| **hash** | 可定义成稳定 | 若可变又参与哈希，会破坏 dict/set 不变式 → **不可哈希**（如 `list`、`dict`） |

---

## 六、字典 key 的最稳记法（含一个常见小坑）

在 Python 里，`dict` / `set` 的元素要满足的是：**可哈希（hashable）**。

- **最稳的一句话**：**能当 key 的东西 = 必须可哈希**（通常也意味着“对该对象来说内容不应变化”）。
- **常见结果**：
  - ✅ 一般可当 key：`str`、`bytes`、数字、`frozenset`、以及**元素全可哈希**的 `tuple`
  - ❌ 一般不能当 key：`list`、`dict`、`set`、`bytearray`、`array.array`（可变 → 默认不提供稳定哈希）

### 小坑：`tuple` 自己不可变，但可能「不可哈希」

`tuple` 是否可哈希取决于它的元素：

```python
a = (1, 2, 3)       # ✅ OK
b = (1, [2, 3], 4)  # ❌ TypeError：tuple 内含 list（不可哈希）
```

直觉：**只要内部藏了可变/不可哈希的东西，整体 hash 就无法稳定**，因此不能当 key。

---

## 七、超级总结（背诵）

- **可变 vs 不可变**：看该类型**是否开放「在同一对象身份下修改内容」的接口**（写入、原地增删等）。  
- **开放了** → 可变；**没开放** → 不可变（想「改」往往只能**新建**再绑定）。  
- **不可变 + 可哈希** → 键友好；**可变** → 一般**不可哈希**当键。

```text
+---------------------------+    +-------------+-------------+
        |                                      |

   通常新建另一对象                        append / []=
```
