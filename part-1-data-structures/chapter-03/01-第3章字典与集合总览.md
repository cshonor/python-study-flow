# 第 3 章「字典和集合」开篇：为什么 dict / set 是基石

---

## 一、开篇核心观点：字典是 Python 的基石

书中有句著名比喻：**Python 是包裹在语法糖里的字典**。这句话不是说“你写的所有代码都是 dict”，而是说：在 Python 的运行时里，**大量机制都可以理解成一张张“名字 → 对象”的映射表**。

你可以把它理解成“查字典”：

- 你写 `x = 1`，其实是在当前命名空间里，把名字 `"x"` 映射到对象 `1`。  
- 你写 `obj.attr`，解释器会在对象/类等相关映射里，按规则查找 `"attr"` 对应的对象。  
- 你写 `f(a=10)`，关键字参数本质上也是“参数名 → 值”的映射（绑定规则在函数调用阶段完成）。

- **语言机制**：类与实例的 `__dict__`、模块命名空间、函数调用时的**关键字参数绑定**，本质都是「名字 → 对象」的映射思路。
- **运行时环境**：内置名可通过 `builtins` 模块访问；教学里常提的 `__builtins__.__dict__` 用来直观看到**内置环境是一张映射表**（具体形态可能因实现细节略有差异，但**命名空间即 dict 类思想**不变）。
- **性能**：CPython 对 `dict` 做了深度优化。多数情况下，`d[key]` 的查找/插入/删除都可以近似理解成 **摊还 O(1)**（不是“永远 O(1)”，但足够快到成为语言的基础积木）。

### 1.1 最小例子：命名空间就是映射（直觉版）

你不需要背实现细节，只要知道“名字查找”这件事在运行时非常像查映射表：

```python
import builtins

print("len in builtins:", "len" in vars(builtins))
```

运行结果：`len in builtins: True`。

#### 大白话解释

1. **`builtins` 模块**  
   装着一批**内置名字**对应的实现：`len`、`print`、`str`、`list`、`dict`……（平时不用 `import` 就能调用 `len()`，是因为解释器会把查找链接到内置环境；这里用 `import builtins` 只是**显式拿到那张表**来看。）

2. **`vars(builtins)`**  
   - `vars(obj)` 在常见对象上返回 `obj.__dict__`（属性字典）。  
   - **模块**的 `__dict__` 就是它的**全局名字空间**：模块里定义的函数、常量、以及解释器挂上去的内置名，名字符串 → 对象，**就是一张映射表**。

3. **`"len" in vars(builtins)`**  
   就是在问：名字表里有没有键 `"len"`？有 → `True`。

#### 等价写法（效果一样）

```python
import builtins

assert "len" in dir(builtins)  # 成立 → dir 列出属性名字符串
assert "len" in builtins.__dict__  # 成立 → 与 vars(builtins) 同一张表
```

#### 小实验：临时换掉内置 `len`，再恢复（直观感受「内置也是可查找的名字」）

下面只在**教学脚本 / REPL**里玩一下即可；**不要在生产代码里**长期改 `builtins`，否则全进程里谁调用 `len` 都会中招。

```python
import builtins

_real_len = builtins.len
try:
    builtins.len = lambda seq: _real_len(seq) + 100  # 故意胡来：在原长度上加 100
    assert len([1, 2, 3]) == 103  # 成立 → 此时全局的 len 已被换掉
finally:
    builtins.len = _real_len  # 必须恢复

assert len([1, 2, 3]) == 3  # 成立 → 恢复后又是正常 len
```

要点：**内置函数也是 `builtins` 名字空间里的一个绑定**；换掉绑定就换行为，`try/finally` 用来保证一定改回去。

## 二、集合（`set` / `frozenset`）的定位

集合的核心价值是两件事：**去重**与**成员测试**。

### 2.1 `set`：可变、唯一、适合“查有没有”

`set` 是“元素不重复”的容器。它最常见的用途不是“存一堆东西”，而是“判断某个东西在不在里面”：

```python
blacklist = {"bot", "spammer", "crawler"}
name = "bot"
print(name in blacklist)  # True
```

当数据量大时，`x in some_set` 往往比 `x in some_list` 快得多，因为 set 的成员测试走的是哈希表。

### 2.2 `frozenset`：不可变集合（因此可哈希）

`frozenset` 是不可变集合。它常用于两类场景：

1. 你想把“一个集合”作为 **dict 的 key**（或放进另一个 set）  
2. 你想表达“这是一组固定不变的值”

```python
key = frozenset({1, 2})
d = {key: "ok"}
print(d)
```

为什么普通 `set` 不行？因为 `set` 可变，内容变了哈希就不稳定，哈希表会崩。

## 三、把 dict / set 当成“哈希表”来理解（你会少踩很多坑）

初学者在第 3 章最容易卡住的点，通常不是“不会写 `d['a'] = 1`”，而是下面这些“为什么”：

- 为什么 dict/set 要求 key/元素“可哈希”？  
- 为什么 `list` 不能当 key？为什么 `tuple` 有时能、有时不能？  
- 为什么 dict 现在“看起来有顺序”，但 set 却“还是没顺序”？  
- 为什么 `d.get(k)` 不报错，而 `d[k]` 会报 `KeyError`？  
- 为什么你在循环里改 dict 会报错？

这一节先把“哈希表直觉”铺好，后面的 API/变体/视图/集合运算你就都能推出来。

---

## 四、关键概念 1：什么叫“可哈希”（hashable）？

你可以把“可哈希”记成一句话：

> **可哈希 = 能稳定地算出 hash 值，并且相等的对象必须有相同的 hash。**

为什么哈希表需要这个条件？

因为 dict/set 不是“逐个比较找目标”，而是先通过 `hash(key)` 快速定位到一个位置（或少量候选），再做相等性比较确认。

### 4.1 最小例子：哪些能当 key？

```python
ok_keys = ["str", 1, 3.14, (1, 2), frozenset({1, 2})]
for k in ok_keys:
    print(k, "-> hashable")

try:
    {[]: 1}
except TypeError as e:
    print("list as key ->", e)
```

### 4.2 最常见坑：`tuple` 不一定可哈希

```python
t1 = (1, 2, 3)
print("hash(t1):", hash(t1))

t2 = (1, [2, 3], 4)
try:
    hash(t2)
except TypeError as e:
    print("hash(t2) ->", e)
```

原因很简单：`tuple` 自己不可变，但它如果装了可变的 `list`，整体就无法保证 hash 稳定。

## 五、关键概念 2：`dict` 的“有序”到底是什么意思？

### 5.1 插入顺序 vs 按键排序（必须分清）

Python 3.7+ 的 `dict` **保留插入顺序**，意思是：你按什么顺序插入键，迭代时通常就按这个顺序出来。

但这不是“按键排序”：

```python
d = {}
d["b"] = 1
d["a"] = 2
print(list(d.keys()))  # ['b', 'a']（插入顺序）
```

如果你要按键排序，需要显式 `sorted(d)` 或 `sorted(d.items())`。

### 5.2 为什么 set 仍然“无序”？

`set` 的语义是集合（数学意义的无序容器）。即使你观察到某一次打印看起来“有顺序”，也不要依赖它。需要稳定顺序时，用 `sorted(s)`。

## 六、关键概念 3：`get` / `[]` / `in` 的差异（新手最常写错）

### 6.1 `d[k]`：缺键就抛 `KeyError`

```python
d = {"a": 1}
try:
    print(d["missing"])
except KeyError as e:
    print("KeyError:", e)
```

这适合“缺键就是异常”的场景，比如你认为数据结构必须完整。

### 6.2 `d.get(k, default)`：缺键返回默认值（不抛）

```python
d = {"a": 1}
print(d.get("missing"))        # None
print(d.get("missing", 0))     # 0
```

它适合“缺键是正常情况”的场景，比如统计、配置读取等。

### 6.3 `k in d`：只检查键是否存在

```python
d = {"a": 1}
print("a" in d)        # True
print(1 in d)          # False（检查的是 key，不是 value）
```

## 七、一个必须知道的限制：不要在遍历 dict 时修改它

你可能会写出这样的代码：

```python
d = {"a": 1, "b": 2}
for k in d:
    if k == "a":
        d["c"] = 3
```

在很多情况下这会触发运行时错误（或产生不可预期行为），因为你在迭代过程中改变了容器结构。正确做法通常是：

- 先遍历一个快照：`for k in list(d): ...`
- 或者收集要修改的键再统一处理
