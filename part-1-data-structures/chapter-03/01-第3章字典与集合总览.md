# 第 3 章「字典和集合」开篇：为什么 dict / set 是基石

---

## 一、开篇核心观点：字典是 Python 的基石

书中有句著名比喻：**Python 是包裹在语法糖里的字典**。这句话不是说“你写的所有代码都是 dict”，而是说：在 Python 的运行时里，**大量机制都可以理解成一张张“名字 → 对象”的映射表**。

你可以把它理解成“查字典”：

- 你写 `x = 1`，其实是在当前命名空间里，把名字 `"x"` 映射到对象 `1`。  
- 你写 `obj.attr`，解释器会在对象/类等相关映射里，按规则查找 `"attr"` 对应的对象。  
- 你写 `f(a=10)`，关键字参数本质上也是“参数名 → 值”的映射（绑定规则在函数调用阶段完成）。

- **语言机制**：类与实例的 `__dict__`、模块命名空间、函数调用时的**关键字参数绑定**，本质都是「名字 → 对象」的映射思路。
- **运行时环境**：内置名可通过 `builtins` 模块访问；教学里常提的 `__builtins__.__dict__` 用来直观看到**内置环境是一张映射表**（具体形态可能因实现细节略有差异，但**命名空间即 dict 类思想**不变）。
- **性能**：CPython 对 `dict` 做了深度优化。多数情况下，`d[key]` 的查找/插入/删除都可以近似理解成 **摊还 O(1)**（不是“永远 O(1)”，但足够快到成为语言的基础积木）。

### 1.1 最小例子：命名空间就是映射（直觉版）

你不需要背实现细节，只要知道“名字查找”这件事在运行时非常像查映射表：

```python
import builtins

print("len in builtins:", "len" in vars(builtins))
```

运行结果：`len in builtins: True`。

#### 大白话解释

1. **`builtins` 模块**  
   装着一批**内置名字**对应的实现：`len`、`print`、`str`、`list`、`dict`……（平时不用 `import` 就能调用 `len()`，是因为解释器会把查找链接到内置环境；这里用 `import builtins` 只是**显式拿到那张表**来看。）

2. **`vars(builtins)`**  
   - `vars(obj)` 在常见对象上返回 `obj.__dict__`（属性字典）。  
   - **模块**的 `__dict__` 就是它的**全局名字空间**：模块里定义的函数、常量、以及解释器挂上去的内置名，名字符串 → 对象，**就是一张映射表**。

3. **`"len" in vars(builtins)`**  
   就是在问：名字表里有没有键 `"len"`？有 → `True`。

#### 等价写法（效果一样）

```python
import builtins

assert "len" in dir(builtins)  # 成立 → dir 列出属性名字符串
assert "len" in builtins.__dict__  # 成立 → 与 vars(builtins) 同一张表
```

#### 小实验：临时换掉内置 `len`，再恢复（直观感受「内置也是可查找的名字」）

下面只在**教学脚本 / REPL**里玩一下即可；**不要在生产代码里**长期改 `builtins`，否则全进程里谁调用 `len` 都会中招。

```python
import builtins

_real_len = builtins.len
try:
    builtins.len = lambda seq: _real_len(seq) + 100  # 故意胡来：在原长度上加 100
    assert len([1, 2, 3]) == 103  # 成立 → 此时全局的 len 已被换掉
finally:
    builtins.len = _real_len  # 必须恢复

assert len([1, 2, 3]) == 3  # 成立 → 恢复后又是正常 len
```

要点：**内置函数也是 `builtins` 名字空间里的一个绑定**；换掉绑定就换行为，`try/finally` 用来保证一定改回去。

## 二、集合（`set` / `frozenset`）的定位

集合的核心价值：**去重**与**成员测试（`x in 集合`）**；底层是哈希表，成员查询在平均意义下可视为 **O(1)**（摊还语义），数据一大就比在 `list` 里线性扫 **O(n)** 省得多。

---

### 2.1 `set`：可变、唯一、适合「查有没有」

- **特性**：元素唯一、**可变**（`add` / `discard` / `remove` 等）、**没有下标**（不能像 `list[i]` 那样按位置取）；**不可哈希**，不能当 `dict` 的 key、也不能放进另一个 `set`。  
- **常见用途**：  
  1. 快速成员测试（黑名单 / 白名单）  
  2. 列表去重（再 `list(...)` 转回列表，若还要**保首次出现顺序**见 `dict.fromkeys`，后文会提）  
  3. 需要频繁增删的「动态集合」

```python
blacklist = {"bot", "spammer", "crawler"}
name = "bot"
assert name in blacklist  # 成立 → 成员测试，平均 O(1)

lst = [1, 2, 2, 3]
unique_lst = list(set(lst))
assert set(unique_lst) == {1, 2, 3}  # 成立 → 去重后元素集合正确
```

---

### 2.2 `frozenset`：不可变集合（因此可哈希）

- **特性**：元素唯一、**不可变**（创建后不能改内容）、**可哈希** → 能当 **`dict` 的 key**，也能作为**另一个 `set` 里的元素**。  
- **常见用途**：  
  1. 把「一组标签 / 权限」当作 **dict 的键**（键语义是「整包 frozenset」，与元素顺序无关）  
  2. 需要把集合放进 `set` 做去重或运算  
  3. 表达**固定不变**的常量集合

```python
key = frozenset({1, 2})
d = {key: "ok"}
assert d[key] == "ok"  # 成立 → frozenset 可当 key

# 普通 set 不能当 dict 的 key（不可哈希）；也不能作为另一个 set 的元素
try:
    bad_key = {1, 2}
    {bad_key: "error"}
except TypeError as e:
    assert "unhashable" in str(e).lower()  # 成立

try:
    {frozenset({1}), {2}}  # 外层 set 里放普通 set → 元素须可哈希
except TypeError as e:
    assert "unhashable" in str(e).lower()  # 成立
```

---

### 2.3 关键区别速览（可背）

| 特性 | `set` | `frozenset` |
| --- | --- | --- |
| 可变性 | 可变（`add` / `remove` 等） | 不可变（只读视图） |
| 可哈希 | ❌ 不可哈希 | ✅ 可哈希 |
| 能否当 `dict` 的 key | ❌ | ✅ |
| 能否放进另一个 `set` | ❌（元素须可哈希） | ✅ |
| 典型场景 | 动态去重、成员测试 | 当键、嵌进 `set`、常量权限包 |

---

### 2.4 两道实操小题（复制运行）

**题 1 — `set` 去重**：把 `raw` 里重复整数去掉，结果用 `set` 表示即可。

```python
raw = [3, 1, 3, 2, 1]
uniq = set(raw)
assert uniq == {1, 2, 3}  # 成立 → 只保留唯一元素
```

**题 2 — `frozenset` 当 `dict` 键**：同一组标签 `{read, write}` 与 `{write, read}` 应命中**同一个**键（集合相等与顺序无关）。

```python
tags_by_group = {
    frozenset({"read", "write"}): "rw",
    frozenset({"read"}): "ro",
}
assert tags_by_group[frozenset({"write", "read"})] == "rw"  # 成立 → 与字面顺序无关
assert tags_by_group[frozenset({"read"})] == "ro"
```

## 三、把 dict / set 当成“哈希表”来理解（你会少踩很多坑）

初学者在第 3 章最容易卡住的点，通常不是“不会写 `d['a'] = 1`”，而是下面这些“为什么”：

- 为什么 dict/set 要求 key/元素“可哈希”？  
- 为什么 `list` 不能当 key？为什么 `tuple` 有时能、有时不能？  
- 为什么 dict 现在“看起来有顺序”，但 set 却“还是没顺序”？  
- 为什么 `d.get(k)` 不报错，而 `d[k]` 会报 `KeyError`？  
- 为什么你在循环里改 dict 会报错？

这一节先把“哈希表直觉”铺好，后面的 API/变体/视图/集合运算你就都能推出来。

---

## 四、关键概念 1：什么叫“可哈希”（hashable）？

你可以把“可哈希”记成一句话：

> **可哈希 = 能稳定地算出 hash 值，并且相等的对象必须有相同的 hash。**

为什么哈希表需要这个条件？

因为 dict/set 不是“逐个比较找目标”，而是先通过 `hash(key)` 快速定位到一个位置（或少量候选），再做相等性比较确认。

### 4.1 最小例子：哪些能当 key？

```python
ok_keys = ["str", 1, 3.14, (1, 2), frozenset({1, 2})]
for k in ok_keys:
    _ = hash(k)  # 能算出 hash，一般就可当 dict 的 key / set 的元素
    print(k, "-> hashable")

try:
    {[]: 1}
except TypeError as e:
    print("list as key ->", e)
    assert "unhashable" in str(e).lower()  # 成立 → list 不可哈希
```

**典型输出**（直接复制运行即可对照；最后一行具体措辞可能随版本略变，但都会包含 `unhashable`）：

```text
str -> hashable
1 -> hashable
3.14 -> hashable
(1, 2) -> hashable
frozenset({1, 2}) -> hashable
list as key -> ... unhashable type: 'list' ...
```

#### 核心结论（可哈希 vs 不可哈希）

- **能当 `dict` 的 key / 能放进 `set`**（须可哈希）的常见类型：`str`、`int`、`float`、`tuple`（**且**元组里每个元素都可哈希）、`frozenset` 等。  
- **不能当 key / 不能放进 `set`**（不可哈希）：`list`、`set`、`dict` 等——**内容可变**时，哈希值若跟着变，哈希表定位会乱套，因此语言直接禁止。

#### 一句话记牢（带补丁）

**「不可变 ≈ 可哈希；可变 ≈ 不可哈希」** 对新手够用好记。  
**补丁**：`tuple` 外壳不可变，但若里面塞了 `list`/`dict`/`set`，**整体仍不可哈希**（见下一节）。

### 4.2 最常见坑：`tuple` 不一定可哈希

```python
t1 = (1, 2, 3)
print("hash(t1):", hash(t1))

t2 = (1, [2, 3], 4)
try:
    hash(t2)
except TypeError as e:
    print("hash(t2) ->", e)
```

原因很简单：`tuple` 自己不可变，但它如果装了可变的 `list`，整体就无法保证 hash 稳定。

### 4.3 对比小练习（复制运行）

用「先试 `hash(x)`」判断能不能当 key，比死记类型名更稳：

```python
def is_hashable(x: object) -> bool:
    try:
        hash(x)
        return True
    except TypeError:
        return False


assert is_hashable("a") and is_hashable(0) and is_hashable((1, 2))  # 成立
assert is_hashable(frozenset({1}))  # 成立
assert not is_hashable([]) and not is_hashable({}) and not is_hashable({1})  # 成立 → list/dict/set 不可哈希
assert not is_hashable((1, []))  # 成立 → tuple 里含 list，整体仍不可哈希
```

## 五、关键概念 2：`dict` 的“有序”到底是什么意思？

### 5.1 插入顺序 vs 按键排序（必须分清）

Python 3.7+ 的 `dict` **保留插入顺序**，意思是：你按什么顺序插入键，迭代时通常就按这个顺序出来。

但这不是“按键排序”：

```python
d = {}
d["b"] = 1
d["a"] = 2
print(list(d.keys()))  # ['b', 'a']（插入顺序）
```

如果你要按键排序，需要显式 `sorted(d)` 或 `sorted(d.items())`。

### 5.2 为什么 set 仍然“无序”？

`set` 的语义是集合（数学意义的无序容器）。即使你观察到某一次打印看起来“有顺序”，也不要依赖它。需要稳定顺序时，用 `sorted(s)`。

## 六、关键概念 3：`get` / `[]` / `in` 的差异（新手最常写错）

### 6.1 `d[k]`：缺键就抛 `KeyError`

```python
d = {"a": 1}
try:
    print(d["missing"])
except KeyError as e:
    print("KeyError:", e)
```

这适合“缺键就是异常”的场景，比如你认为数据结构必须完整。

### 6.2 `d.get(k, default)`：缺键返回默认值（不抛）

```python
d = {"a": 1}
print(d.get("missing"))        # None
print(d.get("missing", 0))     # 0
```

它适合“缺键是正常情况”的场景，比如统计、配置读取等。

### 6.3 `k in d`：只检查键是否存在

```python
d = {"a": 1}
print("a" in d)        # True
print(1 in d)          # False（检查的是 key，不是 value）
```

## 七、一个必须知道的限制：不要在遍历 dict 时修改它

你可能会写出这样的代码：

```python
d = {"a": 1, "b": 2}
for k in d:
    if k == "a":
        d["c"] = 3
```

在很多情况下这会触发运行时错误（或产生不可预期行为），因为你在迭代过程中改变了容器结构。正确做法通常是：

- 先遍历一个快照：`for k in list(d): ...`
- 或者收集要修改的键再统一处理
