# 15.5 在运行时读取类型提示：`__annotations__`、`get_type_hints` 与最佳实践

这一节讨论的是“类型提示除了给静态检查器用，还能不能在运行时派上用场？”  
答案是：**可以，但要用正确的方式读**。

在 FastAPI、Pydantic、依赖注入、工具参数自动校验等场景里，框架往往会在运行时读取你的注解，把它当作“轻量元数据”：

- 生成 API 文档
- 解析请求体/查询参数
- 做参数/返回值的运行时校验（或驱动校验）

---

## 一、类型提示存在哪里？`__annotations__`

Python 会把注解存进 `__annotations__`：

- 函数：`fn.__annotations__`
- 类：`Cls.__annotations__`
- 模块：`module.__annotations__`

其中返回值使用键 `'return'`。

---

## 二、为什么“不要直接依赖 `__annotations__`”？

因为 `__annotations__` 的内容在不同设置下可能有差异，最典型的是：

- 启用 `from __future__ import annotations` 时，注解会以**字符串**形式保存（延迟求值）
- 涉及前向引用、泛型、`|` 联合类型、局部命名空间时，直接读 `__annotations__` 很容易得到“你无法直接用的形式”

结论是：

> **`__annotations__` 是底层存储，不是稳定 API。**  
> 运行时想拿到“可用的真实类型对象”，应该用标准库提供的读取函数。

---

## 三、推荐方式：`typing.get_type_hints()` / `inspect.get_annotations()`

### 1）`typing.get_type_hints(obj)`

适合：

- 你想得到“解析后的类型对象”（包括把字符串前向引用解析掉）
- 你愿意接受它的解析开销（它做的事更多）

### 2）`inspect.get_annotations(obj, eval_str=...)`

在新版本 Python 里更像“底层但稳定”的注解读取工具：

- 可选择是否对字符串注解求值
- 更容易做成你自己项目的统一封装入口

---

## 四、工程化最佳实践：封装统一入口（`Checked` 思路）

如果你项目里真的依赖运行时类型提示（例如工具参数校验、自动绑定等），建议：

- **不要在业务代码里散落 `__annotations__` 读取**
- 写一个小封装函数/基类方法，统一用 `get_type_hints` 或 `inspect.get_annotations` 读取
- 以后 Python 版本演进（注解求值策略变化）时，只改这一处

---

## 五、与“外部数据”边界的关系

牢记一个边界：

- **类型提示 ≠ 运行时校验**

即使你能在运行时读到注解，你仍然需要：

- Pydantic（或同类库）来校验外部输入（HTTP/JSON/LLM 输出）

类型提示更像：

- 驱动/配置校验规则的元信息
