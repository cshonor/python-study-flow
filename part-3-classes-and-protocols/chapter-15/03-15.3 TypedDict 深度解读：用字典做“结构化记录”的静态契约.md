# 15.3 `TypedDict` 深度解读：用字典做“结构化记录”的静态契约

`TypedDict`（PEP 589）解决的是一个非常具体、但在工程里极常见的问题：

> 我想继续用 **dict/JSON 风格的数据**做记录/传输，但希望在开发阶段让类型检查器帮我保证“键名正确、值类型正确、必填字段不缺失”。

它非常适合你在 **AI Agent / 量化 / Web3 / FastAPI** 里处理的这类数据：

- API response / request payload
- LLM 工具调用参数（tool args）
- 行情 tick / bar / 订单回报等“字段固定”的记录

---

## 一、核心结论（必须牢记）

- **`TypedDict` 只服务静态检查**：运行时仍然是普通 `dict`
- **不是数据类**：不会生成属性访问（不能 `obj.title`），只能 `obj['title']`
- **它表达的是“键 → 值类型”的映射契约**，比 `dict[str, Any]` 精确得多

---

## 二、基础用法：定义一个结构化记录（量化 tick 示例）

```python
from typing import NotRequired, TypedDict

class StockTick(TypedDict):
    symbol: str
    price: float
    ts: int
    currency: NotRequired[str]
```

这会让静态检查器知道：

- `tick['symbol']` 一定是 `str`
- `tick['price']` 一定是 `float`
- `currency` 可以没有（可选键）

## 三、运行时行为：它不会帮你“校验数据”

`TypedDict` 的常见误区是把它当成“运行时 schema”：

- `TypedDict(...)` 返回仍是 `dict`
- 键缺失/多余键/值类型错误，运行时**不会**自动报错

因此 `TypedDict` 的最佳位置是：

- **你自己构造/维护的数据结构边界**（内部数据流转）

---

## 四、静态检查能抓住哪些错误？

静态类型检查工具（mypy/pyright/IDE）通常能提前捕获：

- **键名拼写错误**：`tick['prcie']` 直接报错
- **值类型错误**：`price` 传成 `str`、`authors` 传成 `bool` 等
- **缺失必填键**：少了 `symbol/price/ts`
- **额外键**（在“字面量构造”场景里更容易检测）

---

## 五、最大坑：JSON 解析的 `Any` 污染（TypedDict 无法替代运行时校验）

典型问题来自：

```python
import json
whatever = json.loads(data)  # -> Any
```

`Any` 会让静态检查器“放弃追踪”，所以你即使写：

```python
def from_json(data: str) -> BookDict:
    return json.loads(data)
```

也不代表输入真的满足 `BookDict`。

工程结论：

- **内部数据**：`TypedDict` 很好用
- **外部输入（JSON/API/LLM 返回）**：你需要运行时校验（例如 Pydantic / msgspec / 自己写校验）

---

## 六、TypedDict vs dataclass / NamedTuple / pydantic：怎么选？

| 你需要什么 | 更推荐 |
|---|---|
| 只想给现有 dict/JSON 形状加“静态契约” | `TypedDict` |
| 需要 `.field` 访问、默认值、方法、可控不变量 | `dataclass` |
| 需要不可变、轻量、tuple 语义 | `NamedTuple` |
| 需要 **外部数据运行时校验/解析/转换** | `pydantic`（或同类库） |

一句话：

> `TypedDict` 是“静态契约”；Pydantic 是“运行时 schema”。

---

## 七、进阶特性速记

### 1）可选键

- `NotRequired[T]`：单个键可选（更现代）
- `total=False`：让默认所有键都可选（不常用，但某些 patch/update payload 有用）

### 2）继承/复用

可以把公共字段抽到基类 `TypedDict` 里再继承扩展，实现字段复用。

---

## 八、落地模板（你可以直接复用到项目）

### 1）AI Agent：消息/工具调用参数（内部结构约束）

- 用 `TypedDict` 固化工具参数键名，减少“拼写错导致工具调用失败”
- 外部输入（LLM JSON）仍建议用 Pydantic 校验后再转成 TypedDict 形状

### 2）量化：行情/订单/回报

- tick/bar/order/fill 这类结构通常稳定，用 `TypedDict` 能显著减少字段漂移
- 边界层（读 CSV/HTTP API）用运行时校验或解析，再进入内部 `TypedDict` 流转
