# 13.3 两种“协议”：动态协议 vs 静态协议（`typing.Protocol`）

本节要把“协议（protocol）”这个词在 Python 里的两种含义彻底分开：

- **动态协议（dynamic protocol）**：传统 Python 的非正式接口——写在文档里、体现在约定里，运行时按需使用
- **静态协议（static protocol）**：PEP 544 引入的 `typing.Protocol`——给类型检查器看的“结构化接口”

理解这件事，你就能解释：

- 为什么你只写 `__getitem__`，对象就能被 `for` 迭代、被 `in` 查询（动态协议）
- 为什么 `Protocol` 能在不强制继承的情况下，让 mypy/IDE 静态保证“方法形状”匹配（静态鸭子类型）

---

## 一、动态协议：非正式接口（运行时 + 结构）

动态协议的要点：

- **存在于约定/文档**：不需要显式声明“我实现了协议”
- **按需满足**：你用到哪些行为，就要求对象提供哪些方法
- **运行时暴露问题**：缺方法时才会抛 `AttributeError` / `TypeError`

### 例子：只实现 `__getitem__` 也能像序列一样用

只实现 `__getitem__`，很多操作就能工作：

- 索引 `v[0]`
- 迭代 `for x in v`（解释器可通过索引序列协议兜底迭代）
- 成员测试 `'E' in v`（会退化为迭代查找）

这也是为什么书里常说：协议可以“部分实现”。

---

## 二、静态协议：`typing.Protocol`（静态 + 结构）

`Protocol` 的要点（容易被误解的地方也在这里）：

- **它服务于静态检查**：`Protocol` 本质上是给类型检查器表达“需要哪些方法/属性”
- **不要求显式继承**：即使类没有 `class C(MyProto): ...`，只要“结构”满足，类型检查器也会认为它兼容（结构化子类型）
- **也可以显式继承**：显式继承能表达意图，并让 IDE/mypy 更容易定位问题

> 结论：`Protocol` 不是“必须显式 implements 才算实现”，它是 **static duck typing**。

### 运行时检查（可选）：`@runtime_checkable`

默认情况下，`Protocol` 不用于运行时 `isinstance` 检查。

如果你需要**可选**的运行时验证，可以用：

- `@runtime_checkable`

然后做：

- `isinstance(x, MyProto)`

注意：这仍然是基于“有没有这些属性/方法”的一种运行时检查，并不等同于静态检查器对签名/类型的严格分析。

---

## 三、动态协议 vs 静态协议：对比表

| 维度 | 动态协议（传统） | 静态协议（PEP 544 `Protocol`） |
|---|---|---|
| **关注点** | 运行时能不能用 | 类型检查器是否认可 |
| **检查时机** | 运行时 | 静态（mypy/pyright/IDE） |
| **是否需要继承** | 不需要 | 不需要（结构匹配即可）；可显式继承表达意图 |
| **实现要求** | “用到什么实现什么” | 协议里声明的成员需满足（静态层面） |
| **失败形态** | 运行时报错 | 静态报错（运行时不受影响） |
