# 16.1 开篇导读：运算符重载的设计哲学与适用场景

开篇引用 Java 之父 James Gosling 对**运算符重载**的顾虑——在 C++ 中若被滥用，会损害可读性与一致性。Java 因此选择不为用户自定义类型提供中缀运算符重载。

Python 的选择不同：**运算符重载是语言与生态的核心能力之一**。在数值计算、金融精度（`decimal.Decimal`）、科学计算（NumPy/Pandas）以及书中向量等类型上，中缀写法让表达式**贴近数学与领域直觉**。

---

## 一、Python vs Java：同一道“复利”题

### Python（`Decimal` + 中缀）

```python
from decimal import Decimal

interest = principal * ((1 + rate) ** periods - 1)
```

公式结构一目了然；`Decimal` 通过特殊方法参与 `*`、`**`、`+`、`-` 等运算。

### Java（`BigDecimal` + 方法链）

无用户自定义类型的运算符重载时，只能写成 `multiply`、`add`、`pow` 等方法调用，形状上离公式更远。

这对比说明的是：**可读性与领域表达**，而非“哪种语言更好”——而是不同设计权衡。

---

## 二、Python 中运算符重载的本质

运算符在 Python 里对应**数据模型中的特殊方法**，例如：

- `a + b` → `a.__add__(b)`（及可能的 `b.__radd__(a)`）
- `a * b` → `a.__mul__(b)` / `__rmul__`
- `a ** b` → `a.__pow__(b)`

类只要实现（并正确配合）这些方法，就能让实例像内置数值类型一样参与表达式。

---

## 三、核心价值（与你的工程场景）

- **量化 / 金融**：高精度 `Decimal`、自定义“资金/仓位/向量”与公式一致。
- **数据处理**：NumPy 等库的向量化运算建立在运算符与 ufunc 之上。
- **自定义几何/代数类型**：书中 `Vector` / `Vector2d` 的 `+`、标量乘法等。

---

## 四、最佳实践与边界

1. **语义一致**：`+` 表示合并/相加等读者公认的含义，不要写成“神秘副作用”。
2. **优先不可变风格**：像 `tuple` 一样，运算返回新对象，避免原地偷偷改状态（除非类型明确是可变容器语义）。
3. **配合 `NotImplemented`**：无法处理对方类型时返回 `NotImplemented`，让 Python 尝试反向方法或给出清晰 `TypeError`。
4. **克制使用**：业务实体（如 `User + Order`）通常不适合用 `+`；数值、向量、金额等**有数学或领域惯例**的类型更合适。
