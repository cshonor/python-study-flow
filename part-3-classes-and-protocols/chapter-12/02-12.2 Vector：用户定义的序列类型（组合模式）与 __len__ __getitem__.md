# 12.2 `Vector`：用户定义的序列类型（组合模式）与 `__len__` / `__getitem__`

这一节的目标是把 N 维 `Vector` 打造成**不可变扁平序列**：对外“像 `tuple` 一样用”，但语义上仍是“向量”。

本章刻意选择 **组合（composition）**，而不是让 `Vector` 继承第 11 章的 `Vector2d`。

---

## 一、核心设计理念：组合 vs 继承

### 为什么不从 `Vector2d` 继承？

- **维度扩展不自然**：`Vector2d` 的 API 与不变量（x/y）天然是二维语义；继承后再硬扩展到 N 维，往往会把二维细节“泄漏”到通用实现里。
- **存储结构更适合组合**：N 维向量的本质是“一串浮点分量”。把分量存进 `tuple`/`array.array`，让外部通过序列协议访问，会更直接、更通用。

### 组合模式长什么样？

- **内部存储**：用一个扁平的数值容器保存分量（例如 `array('d')`）
- **对外接口**：只要实现序列协议关键方法，外部就能用索引、切片、迭代、拆包等“序列语法糖”
- **不可变**：不提供原地修改；需要“变形”就返回新 `Vector`

---

## 二、应用动机：信息检索的向量空间模型（VSM）

在搜索/信息检索里，一篇文档常被表示成高维向量：

- **维度 N**：词汇表大小（例如 10,000）
- **分量**：词权重（如 TF‑IDF）

比较两篇文档的相似度时，一个经典度量是**余弦相似度**：

\[
\cos(\theta)=\frac{\vec a\cdot \vec b}{\lVert\vec a\rVert\lVert\vec b\rVert}
\]

本书示例的目的不是“卷数学性能”，而是展示：只要你的对象满足协议，很多通用算法就能直接复用（例如把向量当作序列去 `zip`）。

---

## 三、工程建议：真实向量运算请用 NumPy/SciPy

这个 `Vector` 主要用于讲解 Python 数据模型与协议实现：

- 需要大量向量/矩阵计算、NLP、信息检索、科学计算时，生产环境应优先选择 **NumPy / SciPy**（以及相关生态如 `gensim`）。

---

## 四、序列的最小身份证：`__len__` + `__getitem__`

要让 `Vector` 拥有序列行为，最小集合就是：

- `__len__`：让 `len(v)` 有意义
- `__getitem__`：让 `v[i]` / `v[i:j:k]` 有意义

在这个起点上，很多“序列语法糖”会自然出现（迭代、拆包、切片等）。

---

## 五、`__len__`：长度就是维度

`Vector` 是 N 维向量，那么：

- `len(v)` 返回 N

通常只是把内部组件容器的长度透传出去即可。

---

## 六、`__getitem__`：一个方法同时服务索引与切片

关键点在于：`__getitem__` 的参数并不总是 `int`。

当你写：

- `v[0]` → 传入 `0`（`int`）
- `v[1:3]` → 传入 `slice(1, 3, None)`（`slice` 对象）
- `v[1:10:2]` → 传入 `slice(1, 10, 2)`

所以典型实现就是在 `__getitem__` 里分支：

- **index 是 `int`**：返回单个分量（标量）
- **index 是 `slice`**：返回一个新的 `Vector`（保持同类型语义）

这种“切片返回同类型新对象”的约定，是让你的类行为像内置序列的关键之一。

---

## 七、为什么切片应该返回 `Vector`？

对用户来说：

- `v[1:3]` 的结果仍然是一个向量，而不是裸列表/数组

这能保持链式操作与一致抽象：

- `v[1:3][0]` 仍然工作
- 未来你为 `Vector` 增加格式化/哈希/数学运算时，切片结果也自动继承这些能力

---

## 八、不可变性的最低保障（本节小结）

本章的目标是“不可变序列”，因此 `Vector` 至少要做到：

- 初始化完成后，不允许重新绑定实例属性（例如 `_components`）

最小实现方式之一是覆写 `__setattr__`：

- **初始化阶段**允许写入 `_components`
- **初始化之后**任何属性重绑定都抛出 `AttributeError`

这样你就得到了一个“行为上不可变”的序列对象（后续小节还会继续完善不可变语义与协议细节）。
