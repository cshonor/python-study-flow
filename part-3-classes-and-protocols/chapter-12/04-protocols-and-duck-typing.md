# 协议与鸭子类型：Python 的“非正式接口”

这一节是理解 Python 面向对象设计的关键：Python 更常用 **协议（protocol）** 来组织可复用的行为，而不是靠“继承体系”来强行绑定类型。

你可以把协议理解为：

- **一组约定俗成的方法集合**
- **实现了这些方法，就能在对应场景里被当作那类对象使用**

它往往只写在文档里，不需要你显式声明“我实现了某协议”。

---

## 一、什么是协议（Protocol）？

### 1）协议是“行为接口”，不是“名义类型”

在 Python 里，“是不是序列”通常不看继承关系，而看你是否支持序列行为：

- `__len__` → 支持 `len(x)`
- `__getitem__` → 支持 `x[i]` / `x[i:j]`

这就是经典的**序列协议**的核心身份证。

### 2）协议可以按需实现

协议不要求你把“所有相关方法”都实现完。

- 你需要 `for` 循环？实现 `__iter__` 或 `__getitem__`（有其一就可能可迭代）
- 你需要 `len(x)`？实现 `__len__`
- 你需要索引/切片？实现 `__getitem__`

**实现最小集合**往往就能获得大量标准库支持。

---

## 二、什么是鸭子类型（Duck Typing）？

鸭子类型的工程化表达就是：

> 我不关心你“是什么类”，我关心你“能做什么事”。

所以一个对象只要“表现得像序列”，它就能被当作序列使用：

- `len(obj)`
- `obj[i]`
- `obj[i:j]`
- `for x in obj`
- `x, y, z = obj`

---

## 三、示例：`FrenchDeck` 为什么天然是“序列”？

`FrenchDeck` 并没有继承 `list`/`tuple`/`Sequence`，但只要实现 `__len__` 和 `__getitem__`，它就能：

- 被 `len(deck)` 处理
- 支持索引与切片
- 被 `for` 迭代
- 参与大量基于序列的通用算法

这不是“魔法”，而是 Python 数据模型的设计：**协议优先于继承**。

## 四、协议 vs 静态语言接口（直觉对比）

| 维度 | Python 协议（动态） | 静态语言接口（如 Java `interface`） |
|---|---|---|
| **声明** | 通常不声明，只实现方法 | 必须显式 `implements` |
| **约束** | 运行时按需使用（“你用到哪就要求哪”） | 编译期强约束 |
| **实现** | 可以“只实现一部分”以满足某场景 | 往往必须实现接口规定的全部方法 |
| **判断** | 行为是否可用（鸭子类型） | 类型是否匹配（名义类型） |

---

## 五、静态协议：`typing.Protocol`（PEP 544）

Python 仍然是动态语言：运行时不会因为你没“声明实现协议”就禁止你调用方法。

但从 Python 3.8 起，可以用 `typing.Protocol` 为静态类型检查器提供“结构化子类型（static duck typing）”：

- **动态**：照旧靠行为工作
- **静态**：类型检查器可以提前发现“缺方法/签名不匹配”等问题

配套 demo 里会给一个极简 `Protocol`，对比：

- **运行时**：只要对象有 `.speak()` 就能调用
- **静态检查**：只有符合协议的方法签名才算“类型上可用”

## 六、回到第 12 章：为什么 `Vector` 不必继承 `Sequence`

本章的核心练习是让 `Vector` **实现序列协议**：

- `__len__`
- `__getitem__`

实现它们之后，`Vector` 就能自然获得：

- `len(v)`
- `v[i]` / `v[i:j:k]`
- 迭代、拆包、切片等“序列语法糖”

你当然可以选择继承 `collections.abc.Sequence` 来获得一些默认实现（并强化意图表达），但这不是“成为序列”的必要条件。
