# 14.2 `super()`：Python 继承的“灵魂中枢”

本节要解决一个最常见的误解：

> `super()` 不是“父类对象”，也不是“调用父类方法”的语法糖；它是 **按 MRO（方法解析顺序）把调用委托给下一个类** 的机制。

这也是为什么 Fluent Python 强调：

> 坚持使用 `super()` 是确保面向对象 Python 程序可维护性的基本要求。

---

## 一、`super()` 的核心定位：扩展，而不是重写父类逻辑

当子类覆盖父类方法时，最推荐的模式是：

- 让父类先完成它负责的那部分工作
- 子类只做增量扩展

这能避免：

- 复制父类逻辑（容易漂移、难维护）
- 硬编码父类名（多重继承下会破坏协作式调用）

---

## 二、实战：`LastUpdatedOrderedDict`（正确写法）

目标：每次设置键值对后，把 key 移到末尾，形成“按更新顺序”的视图。

关键点是 `__setitem__` 的覆盖：

- 先 `super().__setitem__` 做真实插入/更新
- 再 `move_to_end` 做扩展行为

可运行示例：`super_last_updated_ordereddict_demo.py`。

---

## 三、反例：硬编码父类名的风险

```python
OrderedDict.__setitem__(self, key, value)
```

问题：

- **耦合**：父类名写死，继承结构改动就要全局找/改
- **破坏协作式多重继承**：会绕开 MRO 链路，让“本应被调用的其它基类”失去执行机会

结论：

> 在可预见的未来可能出现多重继承/混入类时，硬编码父类名几乎等价于埋雷。

---

## 四、`super()` 的参数与底层语义

`super()` 的完整形式是：

```python
super(type, obj)
```

在 Python 3 的“零参数 super”里：

- `type` 会被自动推断为当前方法所在的类
- `obj` 会被自动推断为当前的 `self`

**它返回一个代理对象**，会从 `type` 在 `obj.__class__.__mro__` 里的位置出发，找到“下一个类”，并在那条链上解析属性/方法。

所以一句话记忆法是：

> `super()` = “MRO 中当前类的下一个类”，不是“我的父类”。

---

## 五、`__init__`：Python 不会替你自动调用父类初始化

在 Python 里：

- 子类 `__init__` 如果不显式调用 `super().__init__()`，父类初始化就不会发生

这在单继承里会导致“父类字段没初始化”；在多重继承里更会导致协作链路断裂。

---

## 六、多重继承里 `super()` 为什么更重要？

多重继承的正确写法依赖“协作式方法（cooperative methods）”：

- 每个类都调用 `super()....`
- 每个类都只做自己的那部分工作
- MRO 保证每个类以正确顺序被调用一次

可运行示例：`super_mro_diamond_demo.py`（会打印 MRO 和实际调用顺序）。
