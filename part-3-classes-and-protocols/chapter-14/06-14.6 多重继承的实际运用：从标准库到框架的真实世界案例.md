# 14.6 多重继承的实际运用：从标准库到框架的真实世界案例

这一节的核心结论是：

> Python 的多重继承很少用来表达“is-a”，更多用来做 **Mixin 能力拼装**（横向复用）。

因此当你在真实项目里看到多重继承，最常见的形态不是“家族树”，而是：

- 一个“核心基类”（负责主流程）
- 多个 mixin（各自提供单一能力，按 MRO 组合）

---

## 一、多重继承在 Python 的定位

《设计模式》里多重继承几乎只在适配器模式出现；而 Python 的多重继承在工程上真正的用武之地，是：

- **可插拔扩展**（不侵入核心类）
- **按需组合能力**（多个 mixin 叠加）
- **避免深层继承**（用“横向层”替代“纵向层”）

---

## 二、14.6.1 抽象基类（ABC）也是一种 Mixin

很多 `collections.abc` 的抽象基类同时扮演两种角色：

- **接口契约**：规定子类必须实现的抽象方法
- **功能混入**：提供一批可复用的具体方法（默认实现）

例如 `MutableMapping`：

- 抽象方法：`__getitem__` / `__setitem__` / `__delitem__` / `__iter__` / `__len__`
- 具体方法：`get`、`__contains__`、`keys` 等（依赖抽象接口实现）

这种“接口 + 默认方法”的风格，很像 Java 8 的接口默认方法：把“通用逻辑”放在抽象层复用，同时让具体实现只提供最小基元操作。

---

## 三、14.6.2 标准库案例：`ThreadingMixIn`（零侵入扩展）

标准库 `socketserver` 里有两个经典 mixin：

- `ThreadingMixIn`：并发处理请求（多线程）
- `ForkingMixIn`：并发处理请求（多进程）

关键点在于它们的“可插拔”设计：

- 不修改核心的 server 类
- 只通过多重继承把能力混入

在 `http.server` 里你能看到：

- `ThreadingHTTPServer(ThreadingMixIn, HTTPServer)`

配套 demo：`06_real_world_mixins_demo.py`（会实际启动一个线程化 server，并用两个并发请求验证它确实在不同线程处理）。

## 四、14.6.3 框架风格案例：Django 的 CBV mixin（单一职责 + 组合）

Django 的 class-based views（CBV）是 mixin 组合的教科书案例：

- `View`：负责 request dispatch（主流程）
- `TemplateResponseMixin`：负责渲染
- `ContextMixin`：负责上下文
- `MultipleObjectMixin`：负责列表/分页

几个设计点非常值得复用：

- **单一职责**：一个 mixin 只做一件事
- **协作式 super**：每一层都调用 `super()`，把链路交给 MRO 的下一个类
- **可组合**：业务方按需组合 mixin，不改核心流程

我们在 demo 里用纯 Python 写了一个“迷你 CBV”，只保留结构：`View` + 若干 mixin + 最终 concrete view，来展示 MRO 如何把能力拼装起来。

## 五、14.6.4 Tkinter：深 MRO 的历史案例（看懂即可，别模仿）

Tkinter 的组件类层次很深，确实是多重继承的大型实例（例如 `Text` 聚合布局/滚动能力）。

它的教训也很明确：

- 多重继承可以工作，但**深层纵向继承**会带来很高的理解与维护成本
- 现代框架更倾向“组合 + 少量 mixin”，避免过深层次

在 demo 里我们会尝试打印 `tkinter.Text.__mro__`；若运行环境缺少 Tk，则降级提示。

## 六、最佳实践与反模式（本章收尾）

### 最佳实践

- **多重继承主要用于 mixin**，而非表达 is-a
- **mixin 无状态**（或最少状态、并明确约束）
- **方法协作**：尽量使用 `super()` 延续调用链
- **顺序明确**：把“更通用、更接近横切”的 mixin 放在左边，让它先拦截/增强

### 反模式

- 深层纵向多重继承（维护困难）
- 在 mixin 里硬编码父类名（破坏协作式调用）
- 不调用 `super()` 导致链路断裂（后续类失效）
- 直接子类化内置类型试图改变核心语义（见 14.3）
