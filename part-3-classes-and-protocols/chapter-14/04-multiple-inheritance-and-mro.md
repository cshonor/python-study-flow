# 14.4 多重继承与方法解析顺序（MRO）：钻石问题与协作式 `super()`

多重继承的核心难点，不在“能不能继承多个父类”，而在：

- 方法到底会调用哪一个实现？
- 菱形（钻石）结构下，祖先类会不会被调用两次？
- 如何保证每个类都能参与执行、又不重复？

Python 的答案是两件事一起工作：

- **MRO（方法解析顺序，`__mro__`）**：用 C3 线性化算法把继承图变成一条不重复的线
- **协作式 `super()`（cooperative super calls）**：让每一层都“把接力棒交给 MRO 的下一个类”

---

## 一、钻石问题：最小示例

```
      Root
     /    \
    A      B
     \    /
      Leaf
```

疑问是：

- `Leaf().ping()` 时，`Root.ping()` 会不会被调用两次？
- 先走 A 还是先走 B？

在 Python 里，答案取决于 `Leaf(A, B)` 的 **MRO**：

- `Leaf -> A -> B -> Root -> object`

因此只要每一层都用 `super()` 协作式调用，`Root` 只会出现一次。

---

## 二、示例代码：协作方法 vs 非协作方法

### 1）协作的 `ping`

如果 `Leaf/A/B` 的 `ping` 都调用 `super().ping()`：

- 调用链会沿 MRO 依次向后推进
- 直到 `Root.ping()`（再结束）

### 2）非协作的 `pong`

如果某一层（例如 `B.pong`）**不**调用 `super().pong()`：

- 调用链会在这一层终止
- MRO 后面的实现（例如 `Root.pong`）不会被执行

这也是本节最重要的结论之一：

> 在多重继承里，`super()` 不是“父类调用”，而是“继续 MRO 链”；链会不会继续，取决于当前实现是否协作。

---

## 三、`__mro__`：调试多重继承的第一工具

每个类都有 `__mro__`：

```python
print(Leaf.__mro__)
```

遇到“为什么没调用到某个基类方法”的问题时：

1. 先打印目标类的 `__mro__`
2. 再检查链上哪个类没有调用 `super()`

---

## 四、`super()` 的动态性：取决于“你处在哪条 MRO 上”

一个类的方法里写了 `super().ping()`，它调用谁并不只由“这个类继承了谁”决定，而是由 **最终实例的 MRO** 决定。

典型示例是“只在多重继承里才成立的协作类”：

- 类 `U` 自己调用 `super().ping()` 在单独实例化时可能找不到下一个 `ping`
- 但把它放进 `LeafUA(U, A)` 的 MRO 中，就能顺着链路把调用继续交给 `A -> B -> Root`

## 五、真实世界的多重继承：Mixin 组合（预告）

很多标准库/框架的多重继承都不是为了表达 is-a，而是为了“拼装能力”（mixin）。

下一节会专门讨论 Mixin：如何写可组合的 mixin、如何避免状态冲突、以及为什么“协作式 `super()`”几乎是 mixin 的必选项。
