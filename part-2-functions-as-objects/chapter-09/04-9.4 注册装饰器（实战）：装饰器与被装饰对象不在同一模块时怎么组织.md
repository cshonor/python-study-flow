# 9.4 注册装饰器（实战）：装饰器与被装饰对象不在同一模块时怎么组织

## 读完这一节，你只需要带走三句话

1. **`@register` 还是那句老等价**：定义完函数后，立刻执行 **`f = register(f)`**；所以**哪个模块被 import**，哪个模块里的 `@register` 就会跑一遍。  
2. **多模块时谁负责「把人拉进会场」**：你要有一个**明确的地方**去 `import` 那些带 `@register` 的文件；否则它们根本没加载，**注册表里就是空的**。  
3. **注册函数里别干重活**（联网、读盘大文件）：导入会变慢、也容易失败；登记**内存里的名字**就够了。

前面 9.3 已经讲过「为什么一 import 就像执行了代码」；这一节讲**工程里文件怎么摆、`import` 怎么写**，才接得住注册式装饰器。

**配套**：`03_registration.py`、笔记 `03-9.3 注册式装饰器：为什么 import 一下就“执行了代码”.md`。

## 先对齐一下：两种装饰器，别混

### 包装式（常见）：返回一个新函数

调用 `f()` 时，多半先进 **`wrapper`**，再间接调原来的 `f`。

```python
from collections.abc import Callable
from functools import wraps

def log_calls(func: Callable[..., object]) -> Callable[..., object]:
    @wraps(func)
    def wrapper(*args: object, **kwargs: object) -> object:
        print("calling", func.__name__)
        return func(*args, **kwargs)
    return wrapper
```

### 注册式（本节主角）：多半返回「原函数」

重点在**往 `registry` 里记一笔**，不一定改调用方式。

```python
from collections.abc import Callable

registry: list[Callable[[], None]] = []

def register(func: Callable[[], None]) -> Callable[[], None]:
    registry.append(func)
    return func
```

这种写法特别适合：**很多小函数分散在不同文件里，但启动后想统一遍历处理**（路由、子命令、插件、策略表……）。

---

## 工程里为什么经常是「装饰器在 A，函数在 B/C/D」？

因为你需要一个**中心**（注册表）：

- **Web**：把路径和处理函数对上号。  
- **CLI**：子命令名字和处理函数对上号。  
- **插件**：名字和工厂/入口对上号。

注册表可以是 **`list`**（要顺序）、**`dict[str, Callable]`**（要按名查），或更复杂的分组结构——本节不展开数据结构，只讲**文件怎么摆、`import` 怎么触发登记**。

---

## 多模块时：最小可运行的「脑内目录」

假设你的项目里有一个包叫 `demo_pkg`（**把下面所有 `demo_pkg` 换成你真实的包名**），大致长这样：

```text
demo_pkg/
  __init__.py          # 可以是空的，也可以负责“拉齐所有插件”
  registry.py          # 放 registry + register（中心）
  plugins/
    __init__.py        # 常用来 import 本子包里的模块，触发 @register
    alpha.py           # 里面写 @register def ...
    beta.py            # 同上
  bootstrap.py         # 启动入口：先 import，再消费 registry
```

### `registry.py`（中心，只放一份表和装饰器）

```python
from collections.abc import Callable

registry: list[Callable[[], None]] = []

def register(func: Callable[[], None]) -> Callable[[], None]:
    registry.append(func)
    return func
```

### `plugins/alpha.py`（只管写业务函数 + 装饰）

```python
from demo_pkg.registry import register

@register
def alpha_task() -> None:
    print("alpha")
```

### `plugins/beta.py`

```python
from demo_pkg.registry import register

@register
def beta_task() -> None:
    print("beta")
```

### `plugins/__init__.py`（「拉人进会场」的典型写法）

```python
# 只 import 副作用：让 @register 跑起来，把函数塞进 registry
from . import alpha as _alpha  # noqa: F401
from . import beta as _beta    # noqa: F401
```

（`as _alpha` 只是告诉阅读者「我故意不直接用这个名字」；你也可以正常 `import demo_pkg.plugins.alpha`。）

### `bootstrap.py`（启动时先加载，再用表）

```python
import demo_pkg.plugins  # 关键：这一行会间接执行 alpha.py / beta.py 里的 @register

from demo_pkg.registry import registry

for fn in registry:
    fn()
```

**新手最容易漏的一步**：如果你**从来不** `import demo_pkg.plugins`（或里面的 `alpha` / `beta`），那 **`registry` 永远是空的**——不是装饰器坏了，是那些文件**根本没被加载**。

---

## 三种常见「谁来 import」写法（任选一种团队习惯）

1. **启动脚本最开头**：写死 `import myapp.plugins` 或逐个 `import myapp.plugins.foo`。  
2. **包的 `plugins/__init__.py`**：像上面那样把子模块都 import 一遍（集中触发注册）。  
3. **自动发现**（进阶）：用 `importlib` 扫目录按规则加载——新手先别上强度，知道「本质还是 import」即可。

---

## 和「导入即副作用」相关的两个坑（知道就好）

1. **必须保证「该注册的模块会被 import」**——否则表是空的。  
2. **`register` 里别做重活**：导入阶段会跑它；重活放到**明确的启动函数**里（例如 `bootstrap.py` 里 `for fn in registry: ...` 之前再初始化）。
