# 9.3 注册式装饰器：为什么 `import` 一下就“执行了代码”

## 新手大白话：必须死记的一条真相

### 最关键的一句（背下来）

**`@` 装饰器不是在「第一次调用 `f1()`」时才跑，而是在解释器执行到 `def f1` 那一块、完成函数对象创建之后、立刻跑装饰器那一趟。**

在「**把文件当模块 import**」的场景里，等价于：**模块顶层代码从上到下执行一遍**——执行到 `@register` 时，就会**马上**调用 `register(f1)`，**不会**等你以后才 `import` 的某处去 `f1()`。

所以你会感觉：**一 import，怎么装饰器里的 `print` 就出来了？**  
因为：**导入 = 执行模块体**；**装饰器绑定发生在定义阶段**；**函数体里的代码 = 只有调用才跑**。

### 大白话翻译这段等价关系

```python
registry = []

def register(func):
    print(f"running register {func}")
    registry.append(func)
    return func


@register
def f1():
    print("running f1")
```

**`@register` 等价于（在 `def f1` 完成之后立刻执行）：**

```python
f1 = register(f1)
```

重点：

- **`register(f1)` 这一刻就会执行** `print` / `append` —— **不是**等 `f1()`。  
- **`f1` 函数体里的 `print("running f1")`**：只有你写 **`f1()`** 时才会执行。  
- **`if __name__ == "__main__":` 里的代码**：**被 import 时不会跑**（`__name__` 是模块名，不是 `"__main__"`）。

### 和「`self`」一起记

本例**没有 `class`**，全是模块顶层 **`def`** → **谁都不是实例方法** → **自然没有 `self`**。注册式装饰器常见写法就是这样：**普通函数 + 列表/registry**。

### 一眼流程图（执行顺序）

```mermaid
flowchart TD
  A[开始加载模块] --> B[执行 registry = []]
  B --> C[def register 定义完成]
  C --> D[def f1: 创建函数对象]
  D --> E[立刻执行 f1 = register f1]
  E --> F[def f2: 创建函数对象]
  F --> G[立刻执行 f2 = register f2]
  G --> H{__name__ 等于 __main__ 吗}
  H -->|是 直接 python 文件| I[执行 main 里逻辑]
  H -->|否 仅 import| J[到此为止 不进入 main]
```

### 终极四句（肌肉记忆）

1. **`@` 装饰器：在模块加载/执行到定义处时就会跑「装饰这一下」**。  
2. **被装饰函数体：只有调用才跑**。  
3. **`import 模块` = 跑一遍模块顶层**（含 `@`），**不等于**跑 `main()`。  
4. **注册式装饰器**常利用这一点：**import 一次 = 自动登记一堆函数**。

### 超精炼人话版（再压一层）

1. **「导入时完成」是啥**：文件被 **执行** 或 **import** 时，顶层跑到 `@register` + `def f1`，解释器**当场**做完 **`f1 = register(f1)`**——注册、`print`、改名字，**都在这一刻发生**；**不是**等你第一次写 `f1()` 才做。  
2. **你后来 `f1()`**：调用的是**当前名字 `f1` 绑定的那个对象**（本例 `register` 返回原函数，所以体还是那份；若装饰器返回 `wrapper`，你调的就是**加工后的成品**）。  
3. **和你想的一样**：导入阶段就把「该登记的登记好」；后面调用只是**执行函数体**（或先跑 `wrapper` 再间接调原函数）。  
4. **最简一句背**：**加载那一刻 = 装饰/登记做完；`f1()` = 只跑（包装后的）可调用对象。**

- **装饰器在模块导入/执行到函数定义处时就会运行**；
- **函数体不会在导入时执行**，只有你调用函数时才执行；
- `if __name__ == "__main__"` 用来隔离“作为脚本运行”和“作为库被导入”两种场景。

## 一、核心代码（逐行看执行时机）

```python
registry = []

def register(func):
    print(f"running register {func}")
    registry.append(func)
    return func

@register
def f1():
    print("running f1")

@register
def f2():
    print("running f2")

def f3():
    print("running f3")

def main():
    print("running main()")
    print("registry ->", registry)
    f1()
    f2()
    f3()

if __name__ == "__main__":
    main()
```

你可以把 `@register` 读成一条“赋值语句”：

```python
f1 = register(f1)
f2 = register(f2)
```

于是“导入时”就发生了两件事：

- 执行 `register(...)`（打印 + append）；
- 把 `f1` / `f2` 名字重新绑定到装饰器返回值（这里返回原函数，所以行为不变）。

## 二、两种运行场景：脚本运行 vs 作为库导入

### 场景 1：直接运行 `03_registration.py`

```bash
python part-2-functions-as-objects/chapter-09/03_registration.py
```

典型输出顺序（地址会不同）：  

1. `running register ...`（两次）——模块执行到 `@register` 时就触发  
2. `running main()`——因为作为脚本运行，`__name__ == "__main__"` 成立  
3. `registry -> [...]`——注册表里已经有 `f1`、`f2`  
4. `running f1/f2/f3`——函数体只有在被调用时才执行

### 场景 2：仅作为模块被 import（不跑 `main()`）

在 REPL 或其它模块里，需要让 Python 能找到该文件（例如把仓库根目录加入 `PYTHONPATH`，或使用 `importlib` 按路径加载）。**一旦模块体开始执行**，仍会先跑完两次 `running register ...`，然后**不会**进入 `if __name__ == "__main__"` 块。

直觉对比：书里的模块名常写作 `registration`；本仓库脚本文件名为 **`03_registration.py`**，实际 `import` 时的**模块名**取决于你如何把它放进 `sys.path`（不必死记 `import registration` 这一行，抓住「**import 会执行顶层**」即可）。

## 三、结论（你必须形成肌肉记忆的点）

- **装饰器代码（装饰过程）在导入时执行**，所以装饰器可能带来“导入即副作用”。  
- **被装饰函数的函数体不会导入时运行**，只有调用才运行。  
- `if __name__ == "__main__"` 是隔离副作用的基本工具：  
  - 导入时只做定义/注册  
  - 直接运行时才执行“主流程”
