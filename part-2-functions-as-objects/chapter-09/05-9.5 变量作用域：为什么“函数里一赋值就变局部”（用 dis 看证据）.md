# 9.5 变量作用域：为什么“函数里一赋值就变局部”（用 `dis` 看证据）

这一节的目标是解释一个经典坑：

> **只要在函数体里给某名字赋值，Python 就会把它判定为局部变量**（除非 `global` / `nonlocal`）。

这会导致非常困惑的错误：你明明在读全局变量，为什么会 `UnboundLocalError`？

配套脚本：`05_scope_dis_demo.py`。

---

## 一、两个函数：看起来差不多，结果完全不同

```python
a = 10
b = 20

def f1():
    print(a)  # 只读：a 会被当作 global

def f2():
    print(b)  # 你以为读的是全局 b…
    b = 9     # …但这里的赋值让 b 被判定为 local
```

- `f1()` 没有给 `a` 赋值，所以 `a` 是 **global**。
- `f2()` 因为出现了 `b = 9`，所以整个函数体里的 `b` 都被判定为 **local**，包括上面的 `print(b)`。

于是 `print(b)` 执行时，局部变量 `b` 还没绑定，就会触发：

`UnboundLocalError: local variable 'b' referenced before assignment`

---

## 二、怎么修？

如果你确实要在函数里改全局变量，显式声明：

```python
def f2_fixed():
    global b
    print(b)
    b = 9
```

注意：工程里通常更推荐“避免可变全局状态”，但理解 `global` 的语义仍然必要。

---

## 三、用 `dis` 看底层证据：`LOAD_GLOBAL` vs `LOAD_FAST`

Python 在把源码编译成字节码时就决定了某个名字用什么指令加载：

- 读取全局：`LOAD_GLOBAL`
- 读取局部：`LOAD_FAST`

因此对上面的例子：

- `f1` 的字节码会出现 `LOAD_GLOBAL a`
- `f2` 的字节码会出现 `LOAD_FAST b`（因为编译器已把 `b` 判定为局部）

你可以运行配套脚本，看 `dis.dis(f1)` / `dis.dis(f2)` 的差异。
