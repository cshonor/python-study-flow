# 8.6 仅限位置参数与变长参数的类型注解 · 8.7 类型系统的局限性

## 读完这一节，你只需要带走四句话

1. **`/` 左边**：调用时**只能**按顺序塞位置值，**不要**写 `名字=值`。  
2. **`*args: T`**：多出来的**每一个**位置参数，检查器都当你是 **`T`**；在函数体里当**一串同类型的元组**来想。  
3. **`**kwargs: T`**：多出来的关键字参数，**键**在类型里先当 **`str`**；**每一个值**都应是 **`T`**，在函数体里当 **`dict[str, T]`** 来想。  
4. **8.7 收个口**：类型工具再厉害，也**不能代替测试**——业务对不对，还得靠用例和真实数据。

**和第七章的关系**：第七章（尤其 7.7）讲 **`/`、`*args`、`**kwargs` 在运行时怎么调用**；这一节讲**在类型注解里怎么写**，话题连着，但眼睛盯的是 **mypy / Pyright**。

## 8.6 第一部分：`/` —— 「左边别写关键字」

**意思**：`/` **左边**的参数，调用时**只能按位置传**，**不能**写 **`名=值`**。

```python
def f(a: int, /, b: str) -> None: ...
```

- **可以**：`f(1, "x")`  
- **不行**：`f(a=1, b="x")` —— 因为 **`/` 左侧禁止关键字实参**（你会看到解释器报错）。

**人话用处**：参数名可以当「内部实现细节」；以后你想改名，**按位置调用的人**不容易被你改挂。  
**版本**：Python **3.8+** 推荐直接用 **`/`**（PEP 570）。特别老的代码里有时会看到 **`__name` 双下划线前缀** 这种「约定式」写法，新代码不必背，知道有这回事即可。

---

## 8.6 第二部分：`*args` 怎么标类型

**写法**：`*args: T`

**意思**：多出来的**每一个**位置实参，都应该是类型 **`T`**。  
你在函数体里收到的是**元组**；脑子里可以记：**「一串 `T`」≈ `tuple[T, ...]`**。

```python
def func(*args: str) -> None: ...
```

---

## 8.6 第三部分：`**kwargs` 怎么标类型

**写法**：`**kwargs: T`（名字习惯叫 `kwargs`，可以换）

**意思**：多出来的关键字实参，在类型系统里**键先当 `str`**；**每一个值**都应该是 **`T`**。  
脑子里可以记：**`dict[str, T]`**。

```python
def func(**kwargs: int) -> None: ...
```

**一句提醒**：`**kwargs: str | float` 表示的是**「每个值」**要么是 `str` 要么是 `float`**，不是「有的键是这种类型、有的是那种类型」那种复杂形状——那种更适合**改 API**、或用一个**结构写死的字典类型**（例如 **`TypedDict`**，脚本里有 `KLineBar` 小例；更细的用法见 **8.5** 与官方文档）。

## 8.6 第四部分：工作里最常见的「混合模板」——`tag`

```python
def tag(
    name: str,
    /,
    *content: str,
    class_: str | None = None,
    **attrs: str,
) -> str:
    ...
```

用新手能读的方式拆一下：

- **`name` 在 `/` 前**：只能 `tag("img", ...)`，**不要**写 `tag(name="img", ...)`。  
- **`*content: str`**：第一个名字之后的**一串位置正文**，每个都当是 `str`。  
- **`class_`**：在 `*` 后面，可以用关键字传；名字带下划线是因为 **`class` 是 Python 保留字**，不能直接当形参名。  
- **`**attrs: str`**：其它 `xxx=...` 关键字，**值的类型**都应是 `str`。

（实现与 **`07_tag_type_hints_demo.py`** 对齐。）

### 如果「一种 `T`」兜不住真实形状怎么办？

- 常见折中：局部用 **`Any`**，或**把 API 改简单**（少裸 `**kwargs`，改成显式参数、或一个 `dict` / `Mapping`）。  
- 键集合固定、每种键类型不同：考虑 **`TypedDict`**（脚本里 `KLineBar` 看一眼就有数）。

## 8.7：类型系统不是「正确性证明机」

### 1）它保证不了「业务一定对」

mypy / Pyright 主要在看：**形状合不合理、有没有明显的静态矛盾**。  
算法对不对、边界想没想全、需求理解偏没偏——**这些它不管**。

### 2）误报：工具喊错，你心里知道没事

英文叫 **False Positive**。  
常见应对：**把注解写窄一点**、补 **`Protocol`**、升级工具；实在要放行，用**单行** `# type: ignore[具体错误码]` 并**写一句人话注释说明原因**，别整文件瞎忽略。

### 3）漏报：真有问题，工具没吭声

英文叫 **False Negative**。  
常见原因：**`Any` 用太多**、动态 `setattr`、元编程、外面进来的数据没校验就当真……  
应对还是那句老话：**测试**（单元 / 集成）、**边界输入**、该上 **schema / 运行时校验** 就上。

### 4）实践上怎么搭配最省心

- **渐进式**：先把**公共 API**、**最容易出错的热路径**写清楚，别一上来逼自己 100% 铺满。  
- **类型 + 测试**：类型帮你挡一大批低级静态问题；测试帮你挡行为和工具盲区。

## 一眼速查表（想不起来就回来看）

| 写法 | 调用侧直觉 | 在函数体里怎么当 |
|------|--------------|------------------|
| `x: T, /` | `/` 左侧只能位置传 | 就是 `T` |
| `*args: T` | 多出来的位置参数，每个都是 `T` | 当 `tuple[T, ...]` |
| `**kw: T` | 多出来的 `名=值`，每个值都是 `T` | 当 `dict[str, T]` |
| `__x`（旧式） | 老代码里「别关键字传这个」的约定 | 以你用的检查器文档为准 |
