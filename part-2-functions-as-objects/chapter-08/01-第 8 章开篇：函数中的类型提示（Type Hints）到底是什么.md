# 第 8 章开篇：函数中的类型提示（Type Hints）到底是什么

这一章的核心不是“把代码改成静态语言”，而是建立一个边界清晰的认知：

> **类型提示是可选的“静态信息层”，主要给人和工具用；Python 运行时仍然是动态的。**

你可以把类型提示理解成“机器可读的文档/契约”，而不是解释器会强制执行的规则。

配套脚本：`01_type_hints_mypy_demo.py`。

---

## 一、Guido 的态度：加类型提示 ≠ Python 变成静态语言

书里开篇强调的点可以浓缩成三句：

- Python 仍然是动态类型语言；
- 类型提示是**约定**，不是强制；
- 类型提示的价值在于：让工具在“**不运行代码**”的情况下发现问题、提升可读性与协作效率。

这解释了你会在真实项目里看到的现象：

- 代码运行得好好的，但 mypy 报错；
- 或者 mypy 通过了，但运行时仍可能因为逻辑/数据问题出错（类型提示不等于正确性证明）。

---

## 二、历史背景：PEP 484（2014）与“渐进式类型”

PEP 484 提供了给函数参数/返回值标注类型的语法与约定，例如：

```python
def greet(name: str) -> str:
    return "Hello, " + name
```

它的目标不是“让解释器拒绝错误类型”，而是：

- 支持静态分析工具（mypy、pyright、IDE）；
- 让大型代码库的 API 更清晰；
- 允许你“逐步”把类型加进去（**gradual typing**）。

所谓渐进式（Gradual）就是：

- 你可以只给核心模块加；
- 旧代码可以慢慢补；
- 没写注解的部分仍然照常运行、照常动态。

---

## 三、Python 类型系统的风格：鸭子类型与名义类型并存

Python 本来就擅长“鸭子类型”（关注行为，不关注名义类型）：

- 你更关心对象**有没有某个方法/协议**，而不是它是不是某个类的子类。

类型提示并不否定这一点，而是在静态层提供表达方式，例如：

- 用 `typing.Protocol` 描述“像鸭子一样”的接口；
- 用 `Callable` 描述函数/可调用对象；
- 用 `Iterable`/`Sequence`/`Mapping` 描述容器协议。

（这些会在后续章节或第 15 章更系统展开。）  

---

## 四、这一章通常会覆盖什么（你接下来要学的）

结合书的主线，本章通常会围绕“函数”展开：

- 最常见的函数注解写法（参数、返回值）
- `*args` / `**kwargs` 的注解策略
- `Callable[...]` 这类函数类型
- 静态检查工具（mypy）的使用方式与局限

---

## 五、运行证据：类型提示不影响运行，但静态检查会提示

运行这份 demo：

```bash
python part-2-functions-as-objects/chapter-08/01_type_hints_mypy_demo.py
```

你会看到：

- 运行时可以把“错误类型”传进去（解释器不阻止）
- 但在注释里会告诉你：mypy/pyright 会把它标成类型错误

如果你装了 mypy，再跑：

```bash
mypy part-2-functions-as-objects/chapter-08/01_type_hints_mypy_demo.py
```

它会把这些错误指出来（这就是类型提示的实际价值）。
