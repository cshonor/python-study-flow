# 10.1 开篇导读：用一等函数重构设计模式（别被类图绑架）

本章开篇先用一句“反设计模式鸡汤”定调：

> **符合模式并不表示做得对。** —— Ralph Johnson（GoF 之一）

它指向一个事实：**设计模式是解决问题的思路，不是必须照搬的类结构**。当编程语言提供了不同的抽象能力，很多模式的“样板代码”就会被语言特性吞掉。

在 Python 里，最关键的特性是：**函数是一等对象（first-class functions）**。

---

## 一、为什么动态语言会“简化甚至消灭”很多模式

GoF 的 23 个设计模式最初以 C++/Smalltalk 的视角总结，落地到 Java/C++ 时往往需要：

- 接口/抽象类表达“可替换的行为”
- 通过继承/多态组织变化点
- 用对象把“动作”封装起来再传递

而 Python 这类动态语言提供了不同的工具：

- **一等函数**：可以直接传递行为，而不需要为每个策略/命令写一个类
- **闭包**：可以保存状态，把“带状态的对象”压缩成一个函数
- **生成器**：迭代器模式基本变成语法特性
- **鸭子类型**：很多“接口层级”的仪式感被替代为“只要实现需要的方法即可”

经典结论来自 Peter Norvig 的演讲《Design Patterns in Dynamic Languages》：在动态语言里，GoF 里相当多的模式要么消失，要么实现大幅简化。

---

## 二、什么是一等函数（为什么它能替代“类实现的行为对象”）

在 Python 中，函数是普通对象，满足：

- 可以在运行时创建
- 可以赋值给变量
- 可以作为参数传递
- 可以作为返回值
- 可以放进容器（list/dict/set）里统一管理

这意味着：当某个设计模式的核心是“把变化点封装成可替换的行为”，你往往可以：

- **用函数直接表达行为**
- **用字典/列表存放行为**
- **用高阶函数选择/组合行为**

从而把“为了多态而写的类”减少到最小。

---

## 三、本章要做什么（你会看到两种典型模式的 Pythonic 写法）

本章重点会用“一等函数视角”重写两类经典模式：

- **策略模式（Strategy）**  
  目标：把“可替换的算法”抽出来；在 Python 中通常变成“传函数/传可调用对象”。

- **命令模式（Command）**  
  目标：把“要执行的动作”封装成对象以便延迟执行、排队、撤销；在 Python 中常被函数/可调用对象简化（必要时再加少量状态）。

你要训练的能力是：看到一个“模式名”，先问它在解决什么问题，再决定用 Python 的哪种抽象来表达，而不是先画类图。

---

## 四、一个非常实用的提醒

如果你在项目里遇到一段代码“像是设计模式”，可以用这条检查：

- **如果只是为了传递行为**：优先用函数（或 `Callable` / `Protocol` 描述接口）。
- **如果需要保存复杂状态 + 多个协作方法**：用类更清晰。

本章的价值不在于“用函数把所有类都干掉”，而在于：**你有更多选择**，而且能更少地为模式的仪式感付费。
