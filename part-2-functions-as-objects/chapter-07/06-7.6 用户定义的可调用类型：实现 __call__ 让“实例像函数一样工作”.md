# 7.6 用户定义的可调用类型：实现 `__call__` 让“实例像函数一样工作”

7.5 你已经看到：只要 `callable(obj)` 为 True，就能写 `obj(...)`。  
7.6 要做的是把这个能力“掌握在自己手里”：**通过在类里实现 `__call__`，让实例变成可调用对象**。

这在 Python 里特别常见，因为它让你能写出“带状态的函数”：

- 多次调用之间保留状态（计数器、随机抽样器、缓存器）
- 将配置放在 `__init__`，将执行逻辑放在 `__call__`
- 作为装饰器对象（带参数装饰器经常用类 + `__call__` 实现）

配套脚本：`06_user_defined_callable_demo.py`。

## 零、核心笔记（背诵 + 面试；与 **§一～§三** 对照）

### 零·一、一句话

**在类里实现 `__call__` → 该类的实例可以像函数一样用括号调用：`obj(...)`。**  
解释器等价于调用 **`obj.__call__(...)`**。

### 零·二、为什么要用 `__call__`（常见三条）

1. **实例变成可调用对象**：**`callable(obj)`** 为 **`True`**（与 **7.5** 九类里的「实现 `__call__` 的实例」同一档）。  
2. **带状态的“函数行为”**：纯 `def` 不挂闭包时不好持久化数据；对象用属性记住状态（剩余球、计数、缓存等）。  
3. **配置与执行分离**：构造/`__init__` 里放好配置，**`__call__`** 里写“每次被 `()` 触发时要做的事”；**类装饰器**也常用这一套。

### 零·三、BingoCage（书上的经典例子；与脚本一致）

**要做的事**：随机顺序；**抽一个少一个**；抽空 **`LookupError`**；对外既可用 **`cage.pick()`** 也可用 **`cage()`**（`__call__` 委托给 `pick`）。

```python
import random


class BingoCage:
    def __init__(self, items) -> None:
        self._items = list(items)
        random.shuffle(self._items)

    def pick(self):
        try:
            return self._items.pop()
        except IndexError as e:
            raise LookupError("pick from empty BingoCage") from e

    def __call__(self):
        return self.pick()


cage = BingoCage([1, 2, 3])
print(cage.pick())
print(cage())
print(callable(cage))
```

更完整的**抽空循环**见 **`06_user_defined_callable_demo.py`**。

### 零·四、面试常问（简答）

1. **`__call__` 干什么？** 让**实例**具备**函数调用语法**，同时保留**属性状态**。  
2. **和裸函数比？** 裸函数每次调用默认不“自带”可变状态（除非闭包/全局）；**对象 + `__call__`** 把状态放在 **`self`** 上。  
3. **什么时候考虑用？** 需要**跨多次 `()` 调用记住数据**、或要把**配置封进对象再反复执行**、或写**可调用类**（含类装饰器）。

### 零·五、最简模板（背骨架）

```python
class MyCallable:
    def __init__(self) -> None:
        self._state = 0

    def __call__(self, *args, **kwargs):
        self._state += 1
        ...

obj = MyCallable()
obj()
```

### 零·六、三条结论

- **`__call__`**：实例与 **`(...)`** 绑定的入口。  
- **核心价值**：**带状态的可调用对象**。  
- **BingoCage**：**`pick` 做语义，`__call__` 做语法糖**；**`callable(cage) == True`**。

## 一、核心机制：`obj(...)` 到底发生了什么

当你写：

```python
obj(1, 2)
```

如果 `obj` 是一个“实现了 `__call__` 的实例”，解释器会把它转化成：

```python
obj.__call__(1, 2)
```

所以 `__call__` 就是“把对象变成函数”的开关。

---

## 二、BingoCage：一个典型的“带状态可调用对象”

这个例子非常经典，因为它展示了「用对象封装状态」比「用函数反复传参」更自然。

需求：从一组元素中随机抽取，但**抽取一次就消耗一次**，直到抽空为止。

### 2.1 设计要点

- `__init__`：把传入的可迭代对象复制成内部列表，并打乱（防止外部修改影响内部状态）
- `pick()`：弹出一个元素；为空时抛 `LookupError`（语义上比 `IndexError` 更贴合“找不到可抽取项”）
- `__call__()`：把“调用实例”委托给 `pick()`，因此 `cage()` 等价于 `cage.pick()`

---

## 三、为什么不用 `random.choice`

`random.choice(seq)` 每次只是“从序列里选一个”，它**不会消耗元素**，所以会重复选到同一个值；而 BingoCage：

- 内部存储状态（剩余元素）
- 每次抽取会减少剩余集合
- 抽空之后明确报错

这类“需要跨调用保存状态”的问题，正是 `__call__` + 对象封装擅长的。
