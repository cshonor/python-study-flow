# 第 19 章收尾：多核世界中的 Python 并发实践全景（19.7）

## 19.7：免费午餐的终结 & GIL 的历史背景

Herb Sutter《The Free Lunch Is Over》的核心信息可以浓缩成一句话：

- **单线程代码不会再“自动变快”**；想吃多核的红利，你必须引入并发/并行结构。

把时间线放回 Python 的诞生期（90 年代），GIL 更容易理解：

- 当时主流是单核 CPU；GIL 让 **单线程更快、实现更简单、C 扩展更易写**。
- **GIL 是 CPython 的实现权衡**，不是 Python 语言规范；不少语言/实现也做过类似选择。

补充两条非常实用的结论：

- GIL **不妨碍**大量数值/深度学习生态：核心计算常在 C/Fortran/CUDA 中完成，并且能在底层释放 GIL 或绕开 Python 级循环。
- Python 的成功也受益于这种工程权衡：扩展生态繁荣，开发体验好。

## 19.7.1–19.7.3：三大应用场景怎么选

### 1）系统管理 / 运维自动化（I/O 密集为主）

特征：

- SSH/HTTP/文件操作/远程调用多，CPU 计算少

常见选择：

- **线程 / 协程（`asyncio`）**：I/O 等待时能切换，提高吞吐
- 工具生态（如 Ansible/Salt/Fabric 等）本质上就是在规模化 I/O 上做调度

### 2）数据科学 / 机器学习（CPU/GPU 密集为主）

特征：

- 需要吃满多核，甚至 GPU/集群

常见选择：

- **多进程/进程池**：让 Python 层 CPU 密集任务并行起来（绕开 GIL）
- **底层加速库**：NumPy/BLAS、TensorFlow/PyTorch 等多在原生层解决并行
- **并行框架**：例如 Dask 把任务分派到多进程/集群

一句话：数据科学里，线程/协程更多用来做 I/O、数据搬运与管道；核心算力通常由进程/原生库/GPU/分布式承担。

### 3）服务器端 Web / 移动后端（高并发 I/O + 横向扩展）

特征：

- 请求多、等待多（DB/缓存/下游服务），需要水平扩容与清晰分层

落地形态通常是“组合拳”：

- 反向代理/负载均衡（Nginx/Apache）+ 应用服务器 + 数据库/缓存/消息队列
- I/O 并发（异步或多 worker）+ CPU 工作下沉（进程池/任务队列）

## 19.7.4：WSGI/ASGI 与生产部署的并发形态

### WSGI：同步 Web 的标准接口（经典形态）

典型部署结构：

- HTTP 服务器（Nginx/Apache）→ **WSGI 应用服务器**（Gunicorn/uWSGI/mod_wsgi/Unit）→ 多个 Python worker **进程**

要点：

- WSGI 走同步调用，但通过 **多 worker 进程** 把请求分摊到多核上。

### ASGI：面向 `async/await` 的异步接口（新主线）

要点：

- ASGI 允许应用端用协程并发处理 I/O（例如 FastAPI/Starlette 等生态）
- 常见实践依旧是：**多进程 worker + 每个 worker 内事件循环**（以兼顾多核与高并发 I/O）

---

## 19.7.5：分布式任务队列（把“慢活”搬出请求线程）

当 Web 请求里出现这些任务：

- 发邮件、生成报表/PDF、批处理、ETL、图像处理、长耗时调用……

典型做法是把它们从请求路径里剥离：

- **生产者**：Web 请求把任务消息写入队列
- **消费者**：独立 worker 进程/机器从队列取任务执行
- **中间件**：Redis/RabbitMQ/SQS 等负责分发、持久化、重试等能力

常见工具：

- **Celery**（功能全面）
- **RQ**（轻量，基于 Redis）

与 19.6 的关系：

- 19.6 的“队列 + worker + 结束协议”是原型；Celery/RQ 是生产级封装，补上了重试、持久化、监控、路由、定时等工程能力。

## 本章最终选型（可直接记）

| 任务类型 | 首选 | 备注 |
|----------|------|------|
| I/O 密集 | **协程**（`asyncio`）或线程 | 协程并发量更高；线程改造成本更低 |
| CPU 密集（Python 层） | **多进程 / 进程池** | 绕开 GIL，吃多核 |
| 高并发 Web | **ASGI 协程 + 多进程 worker** | I/O 用协程，CPU 下沉到进程池/任务队列 |
| 分布式慢任务 | **任务队列（Celery/RQ）** | 解耦请求与后台作业，便于扩展 |
