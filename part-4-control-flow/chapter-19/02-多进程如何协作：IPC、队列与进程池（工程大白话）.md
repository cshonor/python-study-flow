# 多进程如何协作：IPC、队列与进程池（工程大白话）

**先纠正一个关键认知**：在 CPython 里，**多线程**常被 **GIL** 限制成「同一时刻基本只有一个线程在跑 Python 字节码」——**不等于**你想要的「多核一起算」。  
**真·并行算力**（多核同时跑 Python CPU 循环）通常靠 **多进程**：每个进程**各自一把 GIL**、**内存隔离**。

于是问题变成：**多个并行进程怎么分工、传数据、汇总结果？**  
答案：**不能靠共享全局变量瞎指**，要靠操作系统提供的 **IPC（进程间通信）** 或标准库封装好的 **进程池**。

---

## 一、线程协作 vs 进程协作

| | **多线程** | **多进程** |
|---|------------|------------|
| **内存** | 同进程内共享 | **默认隔离**，各进程一套地址空间 |
| **配合方式** | `list`、`dict`、全局变量 + 锁 | **`Queue` / `Pipe` / 共享内存` / 进程池** |
| **并行感（CPU 纯 Python）** | 往往**吃不满多核** | 可 **多核真并行** |
| **典型用途** | I/O 等待多、改造成本低 | CPU 密集、参数寻优、批量回测子任务 |

---

## 二、多进程常用 4 种配合方式（量化里天天见）

### 1）`Queue`（最常见）

- **主进程**往任务队列塞「子任务」（参数组合、标的、窗口期等）。  
- **多个子进程**各自 `get()` 抢任务，算完把结果 `put()` 进结果队列。  
- **毒药丸**：塞特殊结束标记（如 `None`），让 worker 干净退出。

直觉：**任务分发 + 结果回收**，和「线程池 + 线程安全队列」同一套心智，只是对象换成 **进程安全队列**（`multiprocessing.Queue`）。

### 2）`Pipe`（一对一）

两个进程点对点收发，适合**简单链路**；复杂拓扑一般用队列或池。

### 3）共享内存 `Value` / `Array`（谨慎用）

多进程读写同一块原始内存，**快但容易踩竞态**，要配合锁；适合计数器、简单标志位等，不适合大块复杂结构乱共享。

### 4）进程池 `Pool` / `ProcessPoolExecutor`（最省心）

把「建进程、分任务、收结果、回收」封装掉；你只提交可调用对象与参数。详见 **第 20 章** `concurrent.futures` 与本书 **`06-19.6`** 自建池对照。

## 三、回答你最本质的疑惑

1. **为什么并行进程还能「配合」？**  
   因为**内存虽隔离**，但 OS + 标准库提供了 **IPC**；进程之间通过 **队列/管道/共享内存/套接字** 传**可序列化（pickle）** 的数据，完成分工与汇总。

2. **多线程为啥不能当「多核 Python 计算器」？**  
   线程共享内存很方便，但 **GIL** 下 CPU 型 Python 循环往往**不能线性加速**；I/O 等待场景线程仍然很有用。

---

## 四、一句极简总结

1. **Python 里想多核并行跑 CPU 任务 → 多进程（或进程池）**，别指望多线程「核数翻倍」。  
2. **进程之间默认不能当全局变量用**，要靠 **`Queue` / 池 / Pipe / 共享内存`** 协作。  
3. **量化里经典套路**：主进程拆任务 → 多进程并行算 → 队列或 Future **收结果**（参数寻优、多品种回测子任务都长这样）。
