## 二、GIL 何时「让开」——比「IO 会释放」再细一点

### 1）主动让出（直觉最好记）

常见会**释放 GIL**、让别的线程跑 Python 字节码的路径包括：

- **`time.sleep`**  
- **阻塞式 I/O**：文件、套接字、不少数据库客户端等（等待时线程阻塞，GIL 可让出）  
- 其它进入 **C 层阻塞等待** 的路径（视实现而定）

### 2）时间片与「强制交接」

- 老故事：**3.2 之前**曾以**字节码条数**为粒度做切换；之后主流实现更依赖**基于时间的让渡**思路。  
- 实用：**`sys.getswitchinterval()`** 默认约 **`0.005` 秒（5ms 量级，可调）**——别死记「15ms」这种易过时的数，以你本机读出来的为准。

### 3）C 扩展 / 数值库：量化提速的「暗线」

- **`numpy` / `pandas` / `scipy`** 等大量热点在 **C/Fortran** 里跑，很多路径会 **release GIL** → 单进程里也可能**吃满多核**（尤其在底层 BLAS/OpenMP 已开并行时）。  
- **`numba`** 的 **`njit(parallel=True)`** 等：把 Python 层循环下沉到 LLVM 并行，**常常比「为躲 GIL 盲目多进程」更省 IPC**。  
→ **结论**：看到「CPU 100% 多核」别自动等于「一定用了多进程」；也可能是 **numpy/numba 在 C 层并行**。

## 三、量化里用多进程：三个「致命坑」

### 1）数据拷贝与内存峰值

- **`multiprocessing` 传大对象**往往要 **pickle**；子进程侧再反序列化 → **峰值内存**很容易「看起来只多一份，其实瞬间翻倍」。  
- **大 K 线 `DataFrame` 全表塞进每个任务**：常见 **OOM** 或极慢。  
- **缓解思路（按常见程度）**：  
  - **只传「分片索引 / 路径 / 内存映射句柄」**，让子进程自己读自己那一段；  
  - **`spawn` 启动语义**（尤其 Windows 默认如此）下，注意「父进程状态如何进入子进程」与启动成本；  
  - 需要真共享大块只读数据时，再评估 **`multiprocessing.shared_memory`** / **内存映射文件** 等（复杂度和平台细节都更高）。

### 2）「全局变量」在进程间**不共享**

- 子进程里改 `global x`，**主进程里的 `x` 不会变**——这是**进程隔离**，不是 bug。  
- 要协同状态：用 **`Queue` / `Pipe` / `Value`+`Lock` / `Manager`** 等显式 IPC（见 **02**）。

### 3）GIL 不是「多线程一律垃圾」

- **行情拉取、同步 REST、阻塞 DB**：多线程往往**又轻又够用**；盲目上多进程可能**更慢、更吃内存**。  
- **高并发等待**：再评估 **`asyncio`** 全链路异步化。

## 四、和常见量化工具链的「对齐直觉」

- **参数网格 / 多组合回测**：常见就是 **进程池 + 任务分片 + 结果汇总**（与 **02** 的 `Queue` 模式同构；工程上多用 **`ProcessPoolExecutor`** 封装，见 **第 20 章**）。  
- **Backtrader `optstrategy` 一类思路**（名字因版本而异）：本质仍是「**拆任务 → 并行跑 → 聚合**」，底层逃不开 **进程/执行器 + 序列化成本**。  
- **为什么老手不总用「裸多进程」扛一切**：  
  1. **`numba` / 向量化**：把热点从 Python 层挪走，有时 **单进程就快**。  
  2. **`dask` / `joblib` / 分布式框架**：要的是**任务图 + 调度**，不是手写 `Queue` 一切。  
  3. **向量化回测**：减少 Python 层 `for` 与对象开销，有时比「多进程叠在慢循环上」更划算。

## 五、线程安全 vs 进程安全（风控直觉）

| | **多线程** | **多进程** |
|---|------------|------------|
| **内存** | 共享 | 默认隔离 |
| **风险** | **共享可变状态**不加锁 → 仓位/计数器**错乱** | 不会「互相改对方内存里的 Python 全局变量」 |
| **同步** | `Lock` / `Queue` / 原子结构 | IPC / `Manager` / `Value`+`Lock` |
| **量化直觉** | 单进程实时路径里共享「风控开关」要小心锁 | 跨进程统一限额要**显式**共享与同步，别假装能读「主进程全局变量」 |

## 六、Python 3.13+ 与「无 GIL / free-threaded」一句

- CPython 生态在探索 **free-threaded / 可选无 GIL** 的构建方向（是否安装、默认与否随发行版与平台变化，**以你实际解释器为准**）。  
- **现实**：数值栈与许多扩展的线程安全/ wheel 生态仍在演进；**生产量化**上仍以「**多进程 / C 层并行 / 异步**」为主力更稳妥，别把团队绑在实验构建上。

## 八、量化专用：GIL / 并行 **选型速查表**（直接查）

> **「首选」不是唯一答案**；以**数据量级、是否已有异步库、能否向量化**为准。表内「GIL」指 **CPython 字节码层面**的典型行为。

| 场景 | 首选 | 常见备选 | 关键注意 |
|------|------|----------|----------|
| **单策略纯 Python 双重循环回测** | **`ProcessPoolExecutor` / 多进程分片** | `numba` 并行 / 改向量化 | **少传大 `DataFrame`**；注意 pickle 与内存峰值 |
| **参数网格 / 多标的独立回测** | **进程池 + 分片任务** | `joblib` / `dask` | 任务切分与**结果汇总**；失败重试与日志 |
| **拉行情、REST、阻塞 DB** | **线程池** 或 **`asyncio`（全链路异步）** | 小任务同步也行 | 多进程通常**不划算** |
| **单进程大矩阵因子/截面** | **`numpy`/`pandas`（底层释放 GIL / BLAS 并行）** | `numba` | 留意 **BLAS 线程数**（`OMP_NUM_THREADS` 等）与**过订阅** |
| **海量连接 I/O 等待** | **`asyncio`** | 线程池 | 阻塞调用必须 `to_thread` 或换异步库 |
| **跨进程共享少量整数/标志** | **`multiprocessing.Value` + `Lock`** | `Manager`（更重） | 别把「全局 Python 变量」当共享状态 |
| **实验性 free-threaded 构建** | **观望 / 评估依赖** | 仍用多进程方案 | 生态与 wheel **兼容性**优先 |
