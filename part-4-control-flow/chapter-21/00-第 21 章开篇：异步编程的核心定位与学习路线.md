# 第 21 章开篇：异步编程的核心定位与学习路线

## 异步的“赌徒气质”：为什么说它是双刃剑

异步（async）最容易踩坑的一点是：**你要么彻底避免阻塞，要么收益几乎为零**。

- **优势**：单线程事件循环调度大量 I/O 等待 → 高并发、低开销
- **风险**：只要混进一个阻塞点（同步 I/O、CPU 密集计算、`time.sleep()`…）→ **整个事件循环被卡死**，所有协程一起停摆

因此核心纪律是：

- 异步代码路径里，**阻塞操作必须移出事件循环**（丢给线程/进程），否则 async 架构会失效。

---

## 本章三件事（抓主线）

### 1) 语法四件套

- `async def`：定义协程函数
- `await`：挂起当前协程，把控制权交还事件循环
- `async with`：异步上下文管理器（`__aenter__`/`__aexit__`）
- `async for`：异步迭代（异步迭代器/生成器）

### 2) 核心对象与结构

- **原生协程**：`async def` 返回的协程对象
- **事件循环**：调度中心；只有协程 `await` 时才有机会切换
- **异步上下文/异步迭代**：把“资源管理/迭代协议”扩展到 async 世界

### 3) `asyncio` 与生态

- `asyncio`：标准库事件循环 + 协程调度 + 异步 I/O 基础设施
- 第三方库（如 `httpx`）把网络 I/O 做成可 `await` 的 API，才能真正跑出高并发

---

## 典型场景与边界（最重要的选型点）

- **适合 async**：大量 I/O 并发（HTTP 客户端、爬虫、Web 服务、数据库/消息队列 I/O）
- **不适合直接 async**：CPU 密集计算（会阻塞事件循环）

最佳实践：

- async 负责 I/O 与编排
- CPU 密集 → `asyncio.to_thread` / `run_in_executor` / `ProcessPoolExecutor`（按场景选）
