# 21.11–21.13 Curio、异步原理与核心陷阱（最佳实践清单）

## 21.11：Curio 是什么（为什么书里要提它）

Curio（David Beazley）是一个“从零实现”的异步框架：

- 完全基于 `async/await` 原生协程
- API 倾向于更少的历史包袱、更多的“结构化并发”
- 生态较小，但非常适合用来理解“异步框架真正需要什么”

对你学习的价值是：**看见 asyncio 之外的另一种组织方式**。

---

## Curio vs asyncio（抓最关键差异）

| 维度 | Curio | asyncio |
|------|------|---------|
| 事件循环 | Curio 自己实现（对用户更“隐形”） | 标准库实现（概念与 API 更多） |
| 并发组织 | `TaskGroup`（结构化并发） | `create_task`/`gather`/`as_completed`（更自由也更容易泄露任务） |
| 学习目的 | 更接近“最小必要异步 API” | 生产主流、生态最完整 |

> 注：现代 asyncio 也在吸收结构化并发思想（如 `asyncio.TaskGroup`，Python 3.11+）。

---

## 21.13：异步的底层现实（最重要）

异步不是“让代码并行”，它解决的是：

- 当你在等网络/磁盘时，让 CPU 去做别的事

因此真正的性能收益来自：

- **I/O 等待期间的高并发编排**

而不是让 CPU 密集任务变快。

---

## 核心陷阱（按危害从高到低）

### 1) 把阻塞塞进事件循环

- 同步 I/O（文件、数据库同步驱动、requests、阻塞 DNS…）
- `time.sleep()`
- CPU 密集计算（纯 Python 循环）

后果：**事件循环卡死**，所有协程一起停摆。

解决：

- I/O：用异步库（`httpx.AsyncClient` 等）
- 同步 I/O：`asyncio.to_thread(...)`
- 重 CPU：`ProcessPoolExecutor`（多核并行）

---

### 2) 以为“系统是 I/O 密集型”就万事大吉

更准确的说法是：系统里总会夹杂 CPU 工作与阻塞点。异步的关键是：

- **任何会阻塞 loop 的东西，都必须被隔离/委托**

---

### 3) 任务生命周期失控

典型表现：

- `create_task` 后忘记 `await` 或没收尾
- 取消/超时处理不统一

结构化并发的思路（Curio / 现代 asyncio 的 TaskGroup）就是为了解决这个：**任务在一个作用域里创建，也在这个作用域里结束**。
